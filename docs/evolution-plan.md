# 后续演进规划

## 1. 目的

本文档用于记录 MeshCart 下一阶段值得推进的工程能力、演进顺序和设计决策，避免后续每次做基础设施改造时重新讨论一遍。

当前项目已经具备：

- Gateway + RPC 微服务基本骨架
- `gateway -> user-service` 的 Consul 服务发现
- 基础可观测性链路
- 基于 JWT 的网关登录态

下一阶段建议优先补齐“可持续演进能力”，而不是继续堆新业务接口。

## 2. 优先级建议

建议按以下顺序推进：

1. 运行与排障文档
2. 认证体系升级预案
3. 授权模型预案
4. 服务容器化与本地一键联调
5. 测试与稳定性建设

## 3. 认证体系演进

### 3.1 当前方案

当前项目使用单 token 方案：

- 登录成功后由 `gateway` 签发 JWT
- 受保护接口通过 `Authorization: Bearer <token>` 传递登录态
- 刷新能力依赖当前 JWT 中的时间字段

这个方案适合当前阶段，因为：

- 实现简单
- 接入成本低
- 对现有项目侵入小

### 3.2 未来升级为双 Token

如果后续出现以下需求，应升级为双 token：

- 需要真正的登出即失效
- 需要多端登录管理
- 需要踢人下线
- 需要更短的 access token 生命周期
- 需要降低 token 泄漏后的风险窗口

目标方案：

- `access_token`
  - JWT
  - 有效期短，建议 `15` 到 `30` 分钟
  - 仅承载最小身份信息
- `refresh_token`
  - 建议使用随机高熵字符串
  - 有效期长，建议 `7` 到 `30` 天
  - 服务端维护白名单或会话表

推荐补充字段：

- `access_token`
  - `user_id`
  - `username`
  - `session_id`
  - `iss`
  - `exp`
- `refresh_token` 服务端记录
  - `token_id`
  - `user_id`
  - `session_id`
  - `device_id`
  - `expire_at`
  - `status`

推荐落地方式：

- 使用 Redis 维护 refresh token 白名单
- 每次刷新执行 rotation
- 旧 refresh token 立即作废

建议新增接口：

- `POST /api/v1/user/refresh_token`
- `POST /api/v1/user/logout`
- `POST /api/v1/user/logout_all`

当前不立即升级的原因：

- 现阶段还没有 Redis 会话白名单
- 还没有强会话控制需求
- 单 token 方案足够支撑当前接口联调

## 4. 授权体系演进

### 4.1 当前判断

当前阶段不建议接入 Casbin。

原因：

- 当前项目刚完成认证，还没有稳定的授权模型
- 还没有明确的角色、资源、操作三元关系
- 当前接口规模不足以支撑策略系统的引入成本

### 4.2 什么时候引入 Casbin

当出现以下情况时，再引入 Casbin 更合适：

- 明确需要 RBAC 或 ABAC
- 有后台管理角色体系
- 有多个资源维度需要细粒度控制
- 权限规则需要配置化而不是写死在代码里
- 需要“谁能访问哪个接口/资源”的统一治理

### 4.3 未来推荐方案

引入 Casbin 时建议按以下步骤推进：

1. 先定义角色模型
2. 再定义资源和动作模型
3. 统一网关侧授权拦截点
4. 再接 Casbin policy 存储

建议初始只做 RBAC，不要一开始就上复杂 ABAC。

## 5. 服务容器化与本地联调

当前 `docker-compose.yml` 只提供了可观测性组件和 Consul，本地业务服务仍然主要靠手工启动。

建议下一阶段补齐：

- `gateway` 容器化
- `user-service` 容器化
- MySQL 本地开发容器
- 一键启动依赖与业务链路

这样做的收益：

- 避免宿主机与容器网络地址混用
- 简化 Consul 健康检查配置
- 降低新同学接手项目的启动成本

## 6. 运行手册

当前 [runbook.md](./runbook.md) 还是空的，这个优先级应该比继续加业务接口更高。

建议补齐：

- 本地启动步骤
- Consul / Gateway / user-service 联调步骤
- 常见故障排查
- 日志、指标、链路追踪的查看方式
- JWT 认证问题排查

## 7. 测试与稳定性

当前项目已经有基础结构，但测试覆盖仍然偏弱。

建议逐步补齐：

- `gateway/logic` 单测
- `user-service/biz/service` 单测
- JWT 中间件相关用例
- 登录 / 注册 / 鉴权链路集成测试
- Consul 发现链路的最小验收测试

## 8. 配置与安全

当前配置还偏开发态，下一阶段建议加强：

- 为 `JWT_SECRET` 提供生产环境强制配置
- 避免默认弱密钥进入生产
- 区分开发、测试、生产环境配置
- 明确配置优先级和敏感信息注入方式

## 9. 数据与身份模型

当前登录响应中的 `user_id` 仍可能是占位值，这说明身份模型还没完全打通。

建议后续明确：

- `user-service` 是否要在登录 RPC 中返回真实 `user_id`
- 网关签发 token 时的身份字段口径
- 未来是否要支持 `session_id`
- 是否要支持设备维度登录态

## 10. 当前建议汇总

短期建议先做：

- 补完 `runbook`
- 把 `gateway` 和 `user-service` 放进 Compose
- 明确 `user_id` 返回口径

中期按需推进：

- 双 token + refresh 白名单
- Casbin 授权
- 更完整的测试体系

## 11. 待确认事项

以下事项需要你后续拍板，决定后可以继续补进本文档：

- 是否要优先把业务服务容器化
- 是否要尽快引入 Redis，为双 token 做准备
- 是否会做后台管理角色体系，为 Casbin 做准备
- JWT 默认过期时间是否要从 `120` 分钟缩短
