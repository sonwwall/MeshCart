# 后续演进规划

## 1. 目的

本文档用于记录 MeshCart 下一阶段值得推进的工程能力、演进顺序和设计决策，避免后续每次做基础设施改造时重新讨论一遍。

当前项目已经具备：

- Gateway + RPC 微服务基本骨架
- `gateway -> user-service` 的 Consul 服务发现
- 基础可观测性链路
- 基于 JWT 的网关登录态

下一阶段建议优先补齐“可持续演进能力”，而不是继续堆新业务接口。

## 2. 优先级建议

建议按以下顺序推进：

1. 运行与排障文档
2. 认证体系升级预案
3. 授权模型预案
4. 流量治理与稳定性治理
5. 服务容器化与本地一键联调
6. 运维与发布能力
7. 测试与稳定性建设

## 3. 认证体系演进

### 3.1 当前方案

当前项目使用单 token 方案：

- 登录成功后由 `gateway` 签发 JWT
- 受保护接口通过 `Authorization: Bearer <token>` 传递登录态
- 刷新能力依赖当前 JWT 中的时间字段

这个方案适合当前阶段，因为：

- 实现简单
- 接入成本低
- 对现有项目侵入小

### 3.2 未来升级为双 Token

如果后续出现以下需求，应升级为双 token：

- 需要真正的登出即失效
- 需要多端登录管理
- 需要踢人下线
- 需要更短的 access token 生命周期
- 需要降低 token 泄漏后的风险窗口

目标方案：

- `access_token`
  - JWT
  - 有效期短，建议 `15` 到 `30` 分钟
  - 仅承载最小身份信息
- `refresh_token`
  - 建议使用随机高熵字符串
  - 有效期长，建议 `7` 到 `30` 天
  - 服务端维护白名单或会话表

推荐补充字段：

- `access_token`
  - `user_id`
  - `username`
  - `session_id`
  - `iss`
  - `exp`
- `refresh_token` 服务端记录
  - `token_id`
  - `user_id`
  - `session_id`
  - `device_id`
  - `expire_at`
  - `status`

推荐落地方式：

- 使用 Redis 维护 refresh token 白名单
- 每次刷新执行 rotation
- 旧 refresh token 立即作废

建议新增接口：

- `POST /api/v1/user/refresh_token`
- `POST /api/v1/user/logout`
- `POST /api/v1/user/logout_all`

当前不立即升级的原因：

- 现阶段还没有 Redis 会话白名单
- 还没有强会话控制需求
- 单 token 方案足够支撑当前接口联调

## 4. 授权体系演进

### 4.1 当前判断

当前阶段不建议接入 Casbin。

原因：

- 当前项目刚完成认证，还没有稳定的授权模型
- 还没有明确的角色、资源、操作三元关系
- 当前接口规模不足以支撑策略系统的引入成本

### 4.2 什么时候引入 Casbin

当出现以下情况时，再引入 Casbin 更合适：

- 明确需要 RBAC 或 ABAC
- 有后台管理角色体系
- 有多个资源维度需要细粒度控制
- 权限规则需要配置化而不是写死在代码里
- 需要“谁能访问哪个接口/资源”的统一治理

### 4.3 未来推荐方案

引入 Casbin 时建议按以下步骤推进：

1. 先定义角色模型
2. 再定义资源和动作模型
3. 统一网关侧授权拦截点
4. 再接 Casbin policy 存储

建议初始只做 RBAC，不要一开始就上复杂 ABAC。

## 5. 服务容器化与本地联调

当前 `docker-compose.yml` 只提供了可观测性组件和 Consul，本地业务服务仍然主要靠手工启动。

建议下一阶段补齐：

- `gateway` 容器化
- `user-service` 容器化
- MySQL 本地开发容器
- 一键启动依赖与业务链路

这样做的收益：

- 避免宿主机与容器网络地址混用
- 简化 Consul 健康检查配置
- 降低新同学接手项目的启动成本

## 6. 运行手册

当前 [runbook.md](./runbook.md) 还是空的，这个优先级应该比继续加业务接口更高。

建议补齐：

- 本地启动步骤
- Consul / Gateway / user-service 联调步骤
- 常见故障排查
- 日志、指标、链路追踪的查看方式
- JWT 认证问题排查

## 7. 测试与稳定性

当前项目已经有基础结构，但测试覆盖仍然偏弱。

建议逐步补齐：

- `gateway/logic` 单测
- `user-service/biz/service` 单测
- JWT 中间件相关用例
- 登录 / 注册 / 鉴权链路集成测试
- Consul 发现链路的最小验收测试

## 8. 流量治理

### 8.1 负载均衡

当前项目已经通过 Consul + Kitex resolver 实现基础服务发现，但负载均衡策略还没有单独建模。

后续建议关注：

- 网关到下游 RPC 的实例选择策略
- 权重配置能力
- 故障实例摘除后的流量重分配
- 同机房 / 同可用区优先等就近路由策略

短期建议：

- 先沿用框架默认策略
- 等服务实例数增加后，再评估是否需要显式配置权重和 locality 策略

### 8.2 限流

当前项目尚未接入限流。

后续建议分层处理：

- 网关入口限流
  - 保护整体系统
  - 适合按 IP、用户、接口维度做基础限流
- 下游服务限流
  - 保护热点服务和核心依赖

建议演进顺序：

1. 先做网关级固定窗口或令牌桶限流
2. 再按登录用户或接口维度细化
3. 最后再考虑分布式限流

## 9. 稳定性治理

### 9.1 超时

当前项目需要进一步明确 timeout 策略。

建议统一定义：

- HTTP 请求超时
- Gateway 调下游 RPC 超时
- 下游数据库访问超时

原则：

- 超时必须显式配置
- 越靠近用户入口，超时预算越清晰
- 避免无限等待和层层堆积

### 9.2 重试

当前不建议默认全链路重试。

原因：

- 不区分错误类型的重试会放大故障
- 登录、写操作等非幂等接口盲目重试风险很高

建议：

- 只对幂等读请求或明确可重试的技术错误做有限次数重试
- 把重试和超时一起设计，避免“短超时 + 高重试”制造流量风暴

### 9.3 熔断

当前项目还没有熔断能力。

当下游服务不稳定时，建议未来补齐：

- 网关到下游 RPC 熔断
- 熔断状态指标暴露
- 半开恢复机制

适用场景：

- 下游持续超时
- 错误率持续升高
- 避免故障级联放大

### 9.4 降级

降级能力适合在核心链路之外逐步引入。

建议思路：

- 非核心接口返回兜底结果
- 非关键功能失败时快速返回统一错误
- 在高压场景临时关闭低优先级能力

当前项目可暂不做复杂降级编排，但应该在文档层先保留设计位。

### 9.5 隔离

稳定性治理里最容易被忽视的是隔离。

后续建议关注：

- 不同接口的并发隔离
- 不同下游服务的连接池隔离
- 慢接口和热点接口隔离
- 后台任务与在线请求隔离

如果后续项目规模扩大，隔离优先级会明显上升。

## 10. 运维能力

### 10.1 监控

当前项目已经有 Prometheus + Grafana 基础能力，但仍需补齐指标口径。

后续建议完善：

- 网关 QPS、延迟、错误率
- 下游 RPC 成功率、延迟、错误码分布
- JWT 鉴权失败指标
- Consul 注册与发现异常指标
- 熔断、限流、重试等治理动作指标

### 10.2 链路追踪

当前已经有基础 trace 能力，但后续建议进一步加强：

- 明确 span 命名规范
- 区分业务错误与技术错误标签
- 把认证、鉴权、限流、重试等关键节点纳入 tracing
- 为排障提供“单请求全链路定位”能力

## 11. 发布能力

### 11.1 灰度发布

当前项目还没有灰度发布能力。

未来建议支持：

- 按实例比例灰度
- 按 Header / 用户 / 测试账号灰度
- 灰度指标观测与回滚

这项能力通常和流量治理、动态路由一起建设。

### 11.2 动态路由

未来如果服务数量增加，动态路由会很重要。

可考虑的能力：

- 按请求标签路由到不同版本
- 指定测试流量进入新版本
- 特定租户 / 用户路由到专用实例

在当前阶段不需要立刻实现，但在网关层要保留演进空间。

## 12. 权限控制

权限控制和认证不同，关注的是“用户能做什么”。

当前结论仍然是：

- 现在先不引入 Casbin
- 先把认证链路、受保护路由和身份字段稳定下来

未来如果出现后台管理、资源级权限、多角色体系，再引入 Casbin 更合适。

## 13. 配置与安全

当前配置还偏开发态，下一阶段建议加强：

- 为 `JWT_SECRET` 提供生产环境强制配置
- 避免默认弱密钥进入生产
- 区分开发、测试、生产环境配置
- 明确配置优先级和敏感信息注入方式

## 14. 数据与身份模型

当前登录响应中的 `user_id` 仍可能是占位值，这说明身份模型还没完全打通。

建议后续明确：

- `user-service` 是否要在登录 RPC 中返回真实 `user_id`
- 网关签发 token 时的身份字段口径
- 未来是否要支持 `session_id`
- 是否要支持设备维度登录态

## 15. 当前建议汇总

短期建议先做：

- 补完 `runbook`
- 明确基础 timeout 策略
- 明确限流与错误治理入口放在哪一层
- 把 `gateway` 和 `user-service` 放进 Compose
- 明确 `user_id` 返回口径

中期按需推进：

- 双 token + refresh 白名单
- Casbin 授权
- 网关限流
- 熔断 / 重试 / 隔离
- 灰度发布与动态路由
- 更完整的测试体系

## 16. 待确认事项

以下事项需要你后续拍板，决定后可以继续补进本文档：

- 是否要优先把业务服务容器化
- 是否要尽快引入 Redis，为双 token 做准备
- 是否会做后台管理角色体系，为 Casbin 做准备
- JWT 默认过期时间是否要从 `120` 分钟缩短
- 是否要优先建设网关限流
- 是否要把灰度发布纳入下一阶段目标
