# 高并发治理规划

## 1. 目的

本文档用于基于当前 MeshCart 仓库的实际实现，识别在高并发场景下最需要优先治理的模块、链路和共享状态，并给出建议的推进顺序。

这份文档不讨论“理论上所有系统都可以加的组件”，只讨论当前代码里已经能确认的热点、放大点和风险边界。

## 2. 当前结论

当前项目已经具备一定的稳定性基础：

- `gateway` 已接入请求总超时
- 下游 RPC client 已显式配置 connect timeout / rpc timeout
- `gateway` 已有第一阶段限流
- `inventory-service` 已使用条件更新防止直接超卖
- `order-service` / `payment-service` 已有 action record 幂等骨架

但如果开始进入更真实的并发流量场景，当前最需要优先治理的不是“所有地方一起加组件”，而是以下四类问题：

1. Gateway 多实例后的限流失效
2. 订单 / 支付写链路的幂等收口不够硬
3. 热点 SKU 的库存写竞争
4. Gateway 到下游服务的读流量放大

## 3. 重点风险点



### 3.2 P0：订单 / 支付写链路的幂等还不够硬 #已完成，未测试

当前现状：

- `order-service` 和 `payment-service` 都已经有 action record
- 但它们的幂等前提是调用方提供稳定的 `request_id`
- 当前 `request_id` 还不是所有关键写接口的强约束

影响：

- 用户重复点击、前端重试、网关重放、脚本重复提交时，可能转化成真实多次写入尝试
- 在高并发下，这会放大库存预占、订单创建、支付创建等链路的冲突概率

建议方案：

- 把关键写接口的 `request_id` 从“可选”提升为“强制”
- 至少先覆盖：
  - 创建订单
  - 取消订单
  - 创建支付单
  - 支付确认
- 同时把文档、参数校验和验收测试同步改成强约束

### 3.3 P0：支付创建存在重复活跃支付的并发风险 #已完成

当前现状：

- `payment-service` 创建支付单时，会先查询当前订单已有支付，再决定是否新建
- `payments` 表已增加 `active_order_id` 活跃槽位列
- 已通过数据库唯一索引保证“一个订单同一时刻最多一个 active payment”

影响：

- 如果两个并发请求没有命中同一个 `request_id`
- 且又同时通过“当前没有可复用 active payment”的检查
- 就可能产生多个处于 `pending` 的活跃支付单

已落地方案：

- 新增 `payments.active_order_id`
- `pending` 支付写入 `active_order_id = order_id`
- `succeeded / failed / closed` 会清空 `active_order_id`
- 为 `active_order_id` 建唯一索引，数据库层硬性限制每订单只有一个活跃支付
- `CreatePayment` 在唯一索引冲突时，会回读当前 active payment 并直接复用

当前结论：

- “先查再建”现在只保留为复用优化，不再承担最终并发安全语义
- 最终约束以下沉到数据库唯一索引为准

### 3.4 P1：热点 SKU 会形成库存写热点

当前现状：

- `inventory-service` 预占、释放、确认扣减都是对单个 SKU 库存行做条件更新
- 正确性上已经比“先查后改”稳得多

影响：

- 爆款 SKU 在高并发下会集中打到同一行
- 首先表现为数据库锁竞争、事务等待、RT 抖动
- 后续可能放大成下单超时和库存服务抖动

建议方案：

- 第一阶段先补热点 SKU 指标和锁等待观测
- 第二阶段再评估是否需要：
  - 热点库存隔离
  - 预扣桶 / 分片库存
  - 队列化下单或异步削峰

说明：

- 当前还没有证据说明必须立刻上消息队列
- 更合理的顺序是先补观测，再决定是否需要更重的架构方案

### 3.5 P1：Gateway 到下游读流量存在放大

当前现状：

- Gateway 会频繁调用 `product-service`、`user-service`、`inventory-service`
- 读 RPC 当前已有有限重试，但没有读缓存、熔断和隔离

影响：

- 在高并发或下游轻微抖动时，请求会被重试进一步放大
- 多个读接口叠加时，热点页面和后台接口会先出现 RT 波动

建议方案：

- 先区分“高频读”和“关键写”，不要给写接口盲目加重试
- 对高频读接口评估局部缓存：
  - 商品详情
  - SKU 批量查询
  - 用户基础信息
- 再给关键下游补熔断、失败快速返回和隔离策略

### 3.6 P1：认证 Redis 会成为入口热点之一

当前现状：

- 双 token 已经接入 Redis session store
- 登录、刷新、登出都会访问 Redis
- `refresh_token` rotation 还会走 `WATCH`

影响：

- 当登录 / 刷新流量明显上来时，Redis 会成为入口关键依赖
- 如果缺少连接池、慢查询、失败率和降级指标，排障会比较被动

建议方案：

- 补 Redis 访问指标：
  - 成功率
  - RT
  - 超时
  - 连接池利用率
- 明确 Redis 故障时认证链路的对外表现
- 不建议在没有监控的情况下继续把更多治理能力堆到 Redis 上

## 4. 当前不需要优先做的事

以下能力现在不是最高优先级：

- 全量 MQ 化
- 全链路异步化
- 复杂分布式锁大面积铺开
- 完整 service mesh
- 过早引入大量舱壁 / 线程池 / bulkhead 组件

原因：

- 当前最大问题不是“完全没有组件”
- 而是“多实例入口限流、支付创建唯一性、写接口幂等强约束”还没收口

## 5. 建议推进顺序

建议按以下顺序推进：

### 第一阶段

1. Gateway Redis 分布式限流
2. 强制关键写接口提供 `request_id`
3. 补限流与幂等相关指标、日志、验收测试

### 第二阶段

1. 评估订单维度 / 支付维度的短事务锁
2. 收口高并发下重复创建支付的测试矩阵
3. 视压测结果决定是否需要进一步缩短创建支付事务窗口

### 第三阶段

1. 为库存服务补热点 SKU 指标与锁竞争观测
2. 梳理热点库存的削峰策略
3. 视压测结果决定是否引入更重的库存并发治理

### 第四阶段

1. Gateway 高频读接口局部缓存
2. 下游熔断、失败快速返回、并发隔离
3. Redis / MySQL / RPC 的统一治理看板

## 6. 推荐的第一批落地任务

如果现在开始做，我建议第一批只做下面两件：

### 6.1 Gateway 限流升级为 Redis 分布式限流

原因：

- 改动集中在 `gateway`
- 收益直接
- 一旦上多实例马上有价值

完成标准：

- 登录、注册、刷新、后台写接口全部切到 Redis 限流
- 保留本地内存态实现用于测试和单机开发
- 补 metrics / 日志 / 验收测试

### 6.2 关键写接口强制 `request_id`

原因：

- 这是订单 / 支付并发治理的最低成本护栏
- 不先做这步，后面很多“并发安全”都只是尽量减少损失，不是真正收口

完成标准：

- 创建订单必须带 `request_id`
- 取消订单必须带 `request_id`
- 创建支付必须带 `request_id`
- 支付确认必须带 `request_id`
- gateway 参数校验缺失 `request_id` 直接拒绝
- order-service / payment-service 内部入口仍继续校验 `request_id`
- 接口文档和验收测试同步更新

## 7. 当前建议

当前项目如果开始进入“高并发治理”阶段，最值得先做的不是继续横向铺功能，而是先把下面两件事完成：

1. `gateway` 分布式限流
2. `order / payment` 写接口幂等强约束

在这两项完成之前，不建议过早引入更重的高并发基础设施。

## 8. 核心场景题目解答

本节用于回答当前项目里最常被问到的高并发核心场景问题。

回答原则：

- 已经在项目里落地的，直接说明当前设计
- 还没做的，明确写未来设计思路
- 所有答案都优先结合 MeshCart 当前实现，而不是泛泛而谈

### 8.1 如何防止超卖？

当前已落地：

- `inventory-service` 没有采用“先查库存再扣库存”的裸更新方式
- 预占库存时使用条件更新：
  - `available_stock >= quantity`
  - 原子执行 `reserved_stock + quantity`、`available_stock - quantity`
- 确认扣减时使用条件更新：
  - `reserved_stock >= quantity`
  - 原子执行 `total_stock - quantity`、`reserved_stock - quantity`
- 同时通过 `inventory_reservations` 记录业务预占状态，避免同一业务重复扣减

这套设计已经能防止最直接的超卖。

未来设计思路：

- 如果出现热点 SKU 抢购，再继续补热点库存隔离
- 必要时引入预扣桶、分片库存或队列化削峰
- 在此之前，不建议为了“面试答案完整”就提前消息化

### 8.2 秒杀系统如何设计？

当前项目还没有秒杀专用链路。

未来设计思路：

如果要在当前项目上演进秒杀能力，建议分四层做：

1. 入口限流
   - Gateway 先挡掉超过系统预算的流量
2. 资格校验
   - 用户、活动、时间窗、购买次数限制
3. 库存削峰
   - 热点 SKU 不直接让所有请求同时打数据库
   - 需要按 SKU 维度排队或限并发
4. 订单异步化
   - 秒杀成功先返回“排队中 / 已受理”
   - 后续异步创建订单

当前项目如果真的要上秒杀，第一步不是先写活动页面，而是先补：

- Redis 分布式限流
- 热点库存削峰
- 订单异步受理机制

### 8.3 秒杀系统中如何做限流？

当前已落地：

- `gateway` 已有第一阶段限流
- 但当前实现是单机内存态，只适合单实例开发和基础防护

未来设计思路：

- 秒杀场景限流必须前置到 Gateway
- 优先按三层做：
  - 全局入口限流
  - 活动 / SKU 维度限流
  - 用户维度限流
- 真正进入多实例后，限流存储必须切 Redis 分布式实现

对当前项目来说，秒杀限流的前提条件就是先完成：

- Gateway Redis 分布式限流

### 8.4 秒杀中为什么要用 MQ？

当前项目还没在交易主链路里使用 MQ。

未来设计思路：

- 秒杀时瞬时流量远大于数据库和库存服务的稳定处理能力
- MQ 的核心价值不是“更高级”，而是：
  - 削峰
  - 排队
  - 降低数据库瞬时写压力
  - 把同步强实时下单改成异步受理

对当前项目来说，只有在出现“热点 SKU 的瞬时流量数据库扛不住”时，MQ 才是合理方案。

如果普通交易流量还没到这个级别，先做：

- 分布式限流
- 幂等强约束
- 热点观测

比先引入 MQ 更值钱。

### 8.5 如何应对瞬时高并发流量？

当前项目的建议顺序是：

1. Gateway 限流
2. 写接口幂等
3. 下游超时预算
4. 热点读缓存
5. 必要时异步削峰

原因：

- 真正的问题不是“有没有某个组件”
- 而是异常流量能不能先被挡在系统入口
- 挡不住时，写链路能不能避免重复执行
- 重复执行挡不住时，热点能不能被局部隔离

当前项目里，第一优先级仍然是：

- 把 `gateway` 从单机限流升级到分布式限流

### 8.6 热点商品被大量访问怎么办？

当前已落地：

- 还没有专门的缓存层
- 商品读请求主要通过 Gateway 调 `product-service`

未来设计思路：

优先缓存这些“高频读、低写频、允许短暂旧数据”的对象：

- 商品详情
- 商品列表 / 分类页结果
- SKU 基础映射信息

不建议优先缓存：

- 实时库存
- 订单状态
- 支付状态

因为这些字段的一致性要求更高。

### 8.7 如何设计库存扣减逻辑？

当前已落地：

项目采用的是三段式库存模型：

1. `Reserve`
   - 下单时先预占库存
2. `Release`
   - 取消订单或下单失败时释放预占
3. `ConfirmDeduct`
   - 支付成功后再真正扣减总库存

这套设计的优势：

- 下单成功但未支付时，不会直接扣掉最终库存
- 允许订单超时关闭或取消时释放预占
- 让订单状态和库存状态的协调更清晰

未来设计思路：

- 如果库存热点明显，再增加热点 SKU 削峰策略
- 如果后面引入 MQ，也应该围绕这三段式模型演进，而不是推翻重来

### 8.8 订单和库存如何保证一致？

当前已落地：

- 下单时：
  - 先调用库存预占
  - 再创建订单
  - 如果订单落库失败，会反向释放库存
- 支付成功时：
  - 支付服务先调订单支付确认
  - 订单服务在状态迁移过程中再调用库存确认扣减
- 订单关闭 / 取消时：
  - 订单服务会调用库存释放预占

这说明当前项目采用的是：

- 同步编排
- 局部幂等
- 最终一致

未来设计思路：

- 如果链路继续变长、后续参与方继续增加
- 再考虑把“支付成功”“订单关闭”等动作转成领域事件，通过 MQ 做更彻底的异步一致性

### 8.9 如何保证接口幂等？

当前已落地：

- `order-service` 和 `payment-service` 已有 action record 机制
- 通过 `(action_type, action_key)` 唯一键保证同一动作不被并发重复执行
- 已覆盖创建、取消、支付确认等核心写动作

当前问题：

- 这套机制依赖调用方提供稳定的 `request_id`
- 目前关键写接口还没有全部强制要求 `request_id`

未来设计思路：

- 下一步应把 `request_id` 从可选变成必填
- 对外写接口统一要求客户端或 Gateway 传入幂等键

### 8.10 如何防止重复下单？

当前已落地：

- 如果调用方传入稳定的 `request_id`
- `order-service` 会先查 action record
- 已成功的请求直接返回历史结果
- pending 中的请求直接拦截

当前问题：

- 如果没有 `request_id`
- 用户重复点击或网络重放，系统仍可能进入真实的多次下单尝试

未来设计思路：

- 创建订单必须强制要求 `request_id`
- 前端提交订单时生成请求幂等键
- Gateway / OrderService 都按这个键做幂等收口

### 8.11 如何处理订单支付失败但库存已扣的问题？

当前已落地：

- 当前项目不是“支付创建后立刻扣库存”
- 而是先预占、支付成功后再确认扣减

所以这个问题在当前实现里被拆成两种情况：

1. 支付失败或支付过期
   - 库存仍处于预占态，不是最终扣减态
   - 后续通过订单关闭 / 取消释放库存
2. 支付成功后确认扣减失败
   - 当前会返回错误并阻断状态继续推进
   - 这类场景后续需要更完整的补偿和排障机制

未来设计思路：

- 如果后续链路继续扩展，可以把支付成功后的动作事件化
- 对“已支付但后续扣减异常”这类问题引入异步补偿任务

### 8.12 如何设计一个高并发系统？

结合当前项目，我会把高并发系统设计拆成：

1. 入口治理
   - 限流
   - 超时
   - 熔断
2. 写链路治理
   - 幂等
   - 条件更新
   - 热点隔离
3. 读链路治理
   - 缓存
   - 降级
   - 只读重试
4. 异步化治理
   - 削峰
   - 延迟任务
   - 最终一致

对当前项目来说，最先要做的是：

- 分布式限流
- 幂等强约束

而不是一开始就把系统改成“全异步 + MQ + 分布式锁满天飞”。
