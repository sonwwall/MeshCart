# 分布式事务设计方案

## 1. 文档目标

本文档用于描述 MeshCart 当前针对“商品创建成功但库存初始化失败”这一 `P0` 问题的分布式事务设计方案。

本文档重点回答以下问题：

- 为什么这个问题需要分布式事务收口
- 为什么当前阶段优先选择 `Saga`
- `gateway / product-service / inventory-service` 分别承担什么职责
- 正向流程、补偿流程、异常流程如何设计
- 幂等、空补偿、防悬挂、状态隔离如何处理
- 后续什么时候值得引入 Kafka，以及 Kafka 在这里的作用是什么

## 2. 问题背景

当前商品创建链路如下：

1. `gateway` 调用 `product-service.CreateProduct`
2. `product-service` 落商品和 SKU
3. `gateway` 调用 `inventory-service.InitSkuStocks`
4. `inventory-service` 落库存记录

当前问题在于：

- 如果第 1 步成功、第 2 步成功，但第 3 步失败
- 接口会返回失败
- 但商品和 SKU 实际已经落库

这会造成：

- 接口语义失败，但数据实际部分成功
- 管理端重试可能制造重复商品
- 后续排障和补偿缺少统一口径

因此需要一种机制，保证“商品创建 + 库存初始化”要么整体成功，要么整体回滚。

## 3. 方案选型

当前阶段推荐使用 `Saga`，不优先选择 `TCC`、`XA/2PC`。

### 3.1 为什么选 Saga

原因如下：

- 当前链路只有两个核心参与方：`product-service` 和 `inventory-service`
- 两个动作都属于明确的业务操作，天然适合“正向动作 + 补偿动作”
- 当前系统还没有复杂预留态模型，不适合直接上 `TCC`
- 当前项目处于业务主链路建设阶段，`Saga` 的实现复杂度和收益更匹配

### 3.2 为什么不优先选 TCC

`TCC` 虽然也能解决问题，但当前阶段成本更高：

- 需要给商品和库存都设计 `Try / Confirm / Cancel`
- 需要给商品和库存都引入中间态数据
- 要处理空回滚、防悬挂、确认幂等、取消幂等
- 对现有接口和模型侵入更重

因此在当前阶段不作为首选方案。

### 3.3 为什么不建议用 XA / 2PC

不建议的原因：

- 对数据库和协调器依赖更强
- 阻塞和可用性问题更明显
- 微服务边界下实现成本高
- 后续演进灵活性差

当前问题属于业务事务，不值得为此把系统绑到传统数据库分布式事务模型上。

## 4. 事务目标

本方案的目标是：

1. 商品创建与库存初始化形成统一全局事务
2. 正向步骤全部成功后才对外视为创建成功
3. 任一步失败时，通过补偿动作回滚之前已成功的步骤
4. 全流程必须支持幂等、可观测、可重试

事务目标不是“绝对同步锁死所有参与方”，而是：

- 在业务语义上实现原子性
- 在工程实现上保证失败可补偿、结果可追踪

## 5. 事务边界

当前全局事务只覆盖以下范围：

- `product-service.CreateProduct`
- `inventory-service.InitSkuStocks`

不包含：

- 商品图片上传
- 搜索建索引
- 推荐系统同步
- 审计事件投递
- 其他后续异步扩展动作

原因是当前最需要解决的是“商品主数据和库存主数据的一致性”，不应把后续所有外围动作一次性都拉进事务。

## 6. 参与方职责

### 6.1 `gateway`

`gateway` 负责：

- 接收管理端创建商品请求
- 作为全局事务发起入口
- 调用事务协调器发起 `Saga`
- 传递用户身份、目标商品状态、SKU 初始化信息
- 向调用方返回最终事务结果

`gateway` 不负责：

- 自己维护事务日志
- 自己做本地补偿判定
- 自己持久化商品或库存数据

### 6.2 事务协调器

协调器负责：

- 管理全局事务 ID
- 按顺序调用各参与方正向接口
- 某一步失败时触发已完成步骤的补偿接口
- 记录事务状态
- 支持事务查询、重试和排障

协调器可以采用现成的 Saga 框架或事务中间件实现。

### 6.3 `product-service`

`product-service` 需要提供：

- 正向接口：创建商品和 SKU
- 补偿接口：回滚本次创建的商品和 SKU

`product-service` 要保证：

- 正向接口幂等
- 补偿接口幂等
- 补偿只作用于当前事务创建的数据
- 未确认成功的数据不会被前台当成正常商品长期暴露

### 6.4 `inventory-service`

`inventory-service` 需要提供：

- 正向接口：初始化 SKU 库存
- 补偿接口：回滚本次初始化的库存记录

`inventory-service` 要保证：

- 正向接口幂等
- 补偿接口幂等
- 补偿只作用于当前事务初始化的库存记录

## 7. Saga 正向流程

推荐时序如下：

1. 管理端请求到达 `gateway`
2. `gateway` 生成业务请求并发起全局事务
3. 协调器调用 `product-service.CreateProductSaga`
4. `product-service` 在本地事务中：
   - 创建商品
   - 创建 SKU
   - 记录本次事务关联关系
5. `product-service` 返回 `product_id`、`sku_ids`
6. 协调器调用 `inventory-service.InitSkuStocksSaga`
7. `inventory-service` 在本地事务中：
   - 为每个 `sku_id` 创建库存记录
   - 记录本次事务关联关系
8. 所有步骤成功后，全局事务成功
9. `gateway` 返回创建成功

关键点：

- 各服务内部仍然使用自己的本地事务
- 全局原子性依赖 Saga 的补偿，不依赖单一数据库事务

## 8. Saga 补偿流程

如果库存初始化失败，流程如下：

1. `product-service.CreateProductSaga` 已成功
2. `inventory-service.InitSkuStocksSaga` 失败
3. 协调器触发 `product-service.CompensateCreateProductSaga`
4. `product-service` 回滚本次事务创建的商品和 SKU
5. 全局事务标记为失败
6. `gateway` 返回失败

如果未来出现更长链路，原则仍然相同：

- 后执行的步骤失败
- 协调器按相反顺序触发之前已成功步骤的补偿

## 9. 商品侧补偿设计

### 9.1 补偿目标

商品侧补偿的目标是撤销“本次事务创建的商品主数据”。

### 9.2 当前推荐做法

在当前场景下，补偿优先使用“物理删除本次新建商品和 SKU”。

这样做的前提是：

- 补偿只发生在创建链路当次
- 商品尚未进入后续订单、购物车、审计等引用链路
- 该商品仍然处于创建事务上下文中

原因：

- 当前问题发生在创建接口内部
- 事务失败时不应留下对外不可解释的商品脏数据
- 直接删除比保留一条 `create_failed` 商品更符合“整体失败”的语义

### 9.3 补偿接口建议

建议提供：

- `CreateProductSaga`
- `CompensateCreateProductSaga`

补偿接口入参至少应包含：

- `global_tx_id`
- `product_id`
- `operator_id`

商品服务必须校验：

- 该商品是否由当前事务创建
- 是否已经被补偿过
- 是否已经被其他业务引用

如果已被补偿过，应直接返回成功，保证幂等。

## 10. 库存侧补偿设计

### 10.1 补偿目标

库存侧补偿目标是撤销“本次事务初始化的库存记录”。

### 10.2 当前推荐做法

当前建议优先物理删除本次事务创建的库存记录。

原因：

- 当前库存初始化属于创建链路的一部分
- 事务失败时，这批库存记录不应独立保留
- 这类补偿对象还没有进入预占、扣减、审计等复杂状态机

### 10.3 补偿接口建议

建议提供：

- `InitSkuStocksSaga`
- `CompensateInitSkuStocksSaga`

补偿接口入参至少应包含：

- `global_tx_id`
- `sku_ids`

库存服务必须保证：

- 重复补偿时直接成功
- 没有对应库存记录时不报错
- 只补偿当前事务初始化的数据

## 11. 状态隔离设计

即使采用 Saga，也不建议让商品在事务未完成前直接以目标状态对外暴露。

推荐做法：

- 商品正向创建时，先以内部中间态落库，例如：
  - `draft`
  - `initializing`
- 全局事务成功后，再切换为目标状态
- 如果事务失败，补偿删除商品；若未来不适合删除，再退化为 `create_failed`

当前阶段如果不想扩商品状态枚举，也至少应保证：

- 商品创建事务未完成前不要暴露为 `online`

这样可以避免：

- 补偿尚未完成时用户已看到商品
- 事务窗口内产生额外业务引用

## 12. 幂等设计

分布式事务里幂等不是可选项，而是必须项。

### 12.1 为什么必须幂等

原因包括：

- 协调器可能重试调用正向接口
- 协调器可能重试调用补偿接口
- 网络超时后，调用方无法确认对方是否已成功执行

### 12.2 幂等键建议

建议所有 Saga 接口都带上：

- `global_tx_id`
- `branch_id`

每个参与方维护自己的事务分支日志表，例如：

- `product_tx_branches`
- `inventory_tx_branches`

字段建议：

- `id`
- `global_tx_id`
- `branch_id`
- `action`
- `biz_id`
- `status`
- `created_at`
- `updated_at`

状态建议：

- `prepared`
- `succeeded`
- `compensated`

当同一个 `global_tx_id + branch_id + action` 重复到达时：

- 如果已成功执行，直接返回成功
- 如果已补偿，也要按既定状态返回

## 13. 空补偿与防悬挂

### 13.1 空补偿

空补偿指的是：

- 补偿请求先到
- 正向请求实际上还没成功落库，甚至还没执行

解决思路：

- 补偿接口先查事务分支日志
- 如果正向动作根本没成功，补偿直接返回成功

### 13.2 防悬挂

防悬挂指的是：

- 某分支已经被取消
- 之后迟到的正向请求又执行成功

解决思路：

- 正向接口执行前先检查该事务分支是否已经被标记为补偿完成
- 如果已经补偿，则拒绝继续创建

## 14. 事务日志与排障

要让 Saga 真正可运维，必须有统一的事务可观测性。

建议记录：

- 全局事务 ID
- 每个分支的动作名
- 分支执行状态
- 最后一次错误
- 重试次数
- 发起人
- 商品 ID / SKU ID
- trace_id

排障时至少应能回答：

- 哪一笔创建事务失败了
- 失败卡在哪个分支
- 商品是否已补偿
- 库存是否已补偿
- 是否还能安全重试

## 15. 接口建议

当前建议新增以下内部接口：

### 15.1 `product-service`

- `CreateProductSaga`
  - 正向创建商品和 SKU
- `CompensateCreateProductSaga`
  - 回滚本次事务创建的商品和 SKU

### 15.2 `inventory-service`

- `InitSkuStocksSaga`
  - 正向初始化库存
- `CompensateInitSkuStocksSaga`
  - 回滚本次事务初始化的库存记录

### 15.3 `gateway`

- `POST /api/v1/admin/products`
  - 对外接口不一定改路径
  - 但内部编排改为通过 Saga 协调创建流程

## 16. 推荐时序

### 16.1 成功时序

1. `gateway` 发起全局事务
2. 协调器调用 `product-service.CreateProductSaga`
3. 协调器调用 `inventory-service.InitSkuStocksSaga`
4. 全局事务成功
5. `gateway` 返回成功

### 16.2 失败时序

1. `gateway` 发起全局事务
2. `product-service.CreateProductSaga` 成功
3. `inventory-service.InitSkuStocksSaga` 失败
4. 协调器调用 `product-service.CompensateCreateProductSaga`
5. 全局事务失败
6. `gateway` 返回失败

## 17. 落地顺序建议

建议按以下顺序推进：

1. 先定义事务状态和补偿语义
2. 给商品和库存补事务分支日志表
3. 补商品侧 Saga 正向接口和补偿接口
4. 补库存侧 Saga 正向接口和补偿接口
5. 引入 Saga 协调器
6. 改造 `gateway` 创建商品编排
7. 补幂等、空补偿、防悬挂测试
8. 最后补 runbook 和失败排障文档

## 18. 分布式事务框架选型

当前阶段建议使用分布式事务框架，不建议自己手写事务协调器。

### 18.1 为什么建议使用框架

如果不使用框架，团队需要自己实现以下能力：

- 全局事务 ID 管理
- 分支调用编排
- 分支补偿顺序管理
- 分支重试与超时处理
- 事务状态查询
- 幂等与空补偿协作约定
- 失败排障入口

这些能力本身就已经是一套基础设施。  
对当前阶段的 MeshCart 来说，自己实现这套能力的成本和风险都偏高，不值得重复造轮子。

因此当前建议：

- 业务逻辑自己实现
- 全局事务协调交给成熟框架

### 18.2 推荐框架：DTM

当前最推荐的是 `DTM`。

推荐原因如下：

- Go 生态里对 `Saga / TCC / XA` 支持较成熟
- 当前场景以 `Saga` 为主，DTM 与这个模式比较匹配
- 对 HTTP / gRPC 的接入方式较直接
- 能满足当前“商品服务 + 库存服务”的双服务事务链路
- 后续如果事务链路变复杂，也保留继续演进到 `TCC` 的空间

### 18.3 为什么当前不建议自己写协调器

自己写协调器最大的问题不是“写不出来”，而是：

- 很难一次性把幂等、补偿、重试、查询都做对
- 后续维护成本高
- 团队会长期背一套非标准基础设施
- 事务协调 bug 的排查成本极高

当前更合理的选择是：

- 使用成熟事务框架做协调
- 把精力放在商品和库存两个业务参与方的接口语义上

## 19. 基于 DTM 的落地设计

### 19.1 整体接入方式

当前建议：

- `gateway` 作为 Saga 发起方
- `DTM` 作为事务协调器
- `product-service` 和 `inventory-service` 作为 Saga 分支参与方

整体流程如下：

1. 管理端请求到达 `gateway`
2. `gateway` 创建 Saga 事务
3. `gateway` 向 `DTM` 注册两个分支：
   - 商品正向创建 / 商品补偿
   - 库存正向初始化 / 库存补偿
4. `DTM` 依次调用各分支
5. 某一步失败时，`DTM` 自动触发已成功分支的补偿
6. `gateway` 根据全局事务结果返回成功或失败

### 19.2 当前推荐的服务职责

#### `gateway`

负责：

- 构造 Saga 请求
- 组织每个分支的正向 URL 和补偿 URL
- 传递 `operator_id`、目标商品状态、SKU 初始化信息
- 接收事务结果并对外返回

不负责：

- 自己维护全局事务状态机
- 自己轮询分支状态

#### `DTM`

负责：

- 生成全局事务 ID
- 管理 Saga 分支调用顺序
- 记录全局事务状态
- 某分支失败时触发补偿
- 对失败事务执行重试策略

#### `product-service`

负责：

- 实现商品正向创建分支
- 实现商品补偿分支
- 维护本地事务分支幂等记录

#### `inventory-service`

负责：

- 实现库存正向初始化分支
- 实现库存补偿分支
- 维护本地事务分支幂等记录

### 19.3 DTM 下的分支设计

建议分成两个 Saga 分支：

#### 分支一：商品创建

正向动作：

- `POST /rpc/saga/product/create`

补偿动作：

- `POST /rpc/saga/product/create/compensate`

正向职责：

- 创建商品
- 创建 SKU
- 记录本次事务和 `product_id / sku_ids` 的映射

补偿职责：

- 删除本次事务创建的商品和 SKU
- 或在后续不允许物理删除时，收口为失败态

#### 分支二：库存初始化

正向动作：

- `POST /rpc/saga/inventory/init`

补偿动作：

- `POST /rpc/saga/inventory/init/compensate`

正向职责：

- 为指定 `sku_ids` 初始化库存记录

补偿职责：

- 删除本次事务初始化的库存记录

### 19.4 `gateway` 发起 Saga 的推荐方式

建议在 `gateway` 创建商品逻辑中：

1. 先校验创建商品请求
2. 构造 Saga 分支参数
3. 向 `DTM` 提交 Saga
4. 等待提交结果
5. 根据事务结果返回创建成功或失败

`gateway` 侧最核心的改造是：

- 现有同步顺序 RPC 调用
  - `CreateProduct`
  - `InitSkuStocks`
- 改为通过 `DTM Saga` 统一编排

### 19.5 分支入参建议

#### 商品分支正向入参

- `global_tx_id`
- `operator_id`
- `target_status`
- 商品基础信息
- SKU 列表

#### 商品分支补偿入参

- `global_tx_id`
- `operator_id`
- `product_id`

#### 库存分支正向入参

- `global_tx_id`
- `operator_id`
- `sku_ids`
- `initial_stocks`

#### 库存分支补偿入参

- `global_tx_id`
- `sku_ids`

### 19.6 本地事务分支日志表建议

即使使用 DTM，各参与方仍需要本地分支日志表。

建议商品服务增加：

- `product_tx_branches`

建议字段：

- `id`
- `global_tx_id`
- `branch_action`
- `biz_type`
- `biz_id`
- `status`
- `payload`
- `created_at`
- `updated_at`

库存服务同理增加：

- `inventory_tx_branches`

用途包括：

- 接口幂等判断
- 空补偿判断
- 防悬挂判断
- 事务排障

### 19.7 幂等策略

在 DTM 下，仍然不能把幂等完全交给框架。

业务参与方需要自己保证：

- 同一个 `global_tx_id + branch_action` 重复执行不重复落库
- 补偿重复调用时直接成功
- 已补偿事务的迟到正向请求被拒绝

建议处理方式：

1. 正向请求到达时，先查本地分支日志
2. 如果已成功执行，直接返回成功
3. 如果已补偿完成，拒绝再执行正向动作
4. 补偿请求到达时，如果正向根本未成功，则按空补偿返回成功

### 19.8 商品与库存的当前补偿口径

当前阶段推荐补偿口径如下：

- 商品分支补偿：
  - 删除本次事务创建的商品和 SKU
- 库存分支补偿：
  - 删除本次事务创建的库存记录

这样处理的原因是：

- 问题发生在创建链路当次
- 事务失败时不应留下半成品主数据
- 当前阶段创建链路还没有进入订单引用

如果未来业务约束变化，再把补偿策略收紧为：

- 商品改为 `create_failed`
- 库存改为 `frozen`

### 19.9 DTM 接入后的运行依赖

引入 DTM 后，运行环境需要新增：

- `DTM server`
- 对应的配置项
- `gateway` 到 `DTM` 的访问配置
- 本地 / 测试环境启动方式

因此在工程上还需要同步补：

- 配置文件
- 本地启动文档
- 健康检查与排障说明
- 集成测试

### 19.10 当前阶段的结论

如果 MeshCart 现在要真正落地 Saga 分布式事务，推荐方案是：

- 使用 `DTM`
- 不自己实现事务协调器
- 把 `gateway` 改为 Saga 发起方
- 给 `product-service` 和 `inventory-service` 各自补正向分支与补偿分支
- 通过本地分支日志完成幂等、空补偿、防悬挂

## 20. Kafka 的远期规划

### 20.1 当前阶段为什么不必引入 Kafka

当前这个 `P0` 的核心问题是：

- 两个服务之间的创建动作要么一起成功，要么一起失败

这本质上是事务协调问题，不是事件广播问题。

因此在当前阶段：

- 引入 Kafka 不是解决 Saga 的前置条件
- 不应为了修这个 `P0` 单独引入 Kafka

否则会额外引入：

- 消息生产与消费运维成本
- 消息重复消费治理
- 幂等消费治理
- 死信、积压、回放处理

当前收益不足以覆盖复杂度。

### 20.2 什么时候应该引入 Kafka

当系统出现以下特征时，再考虑引入 Kafka 更合适：

1. 一个商品事件需要被多个下游同时消费
2. 你希望把同步 RPC 编排改成事件驱动解耦
3. 商品、库存、搜索、推荐、审计、风控等模块都需要订阅统一事件流
4. 需要削峰、缓冲、回放和异步扩展能力

典型信号包括：

- 商品创建后不止库存服务一个下游
- 商品变更后还要同步搜索索引、缓存、审计日志
- 事件消费者数量持续增加
- 同步链路越来越长，入口接口延迟明显上升

### 20.3 Kafka 的作用

Kafka 在这个系统里更适合作为“事件总线”，而不是“分布式事务协调器”。

引入后的主要作用包括：

- 发布商品领域事件
  - `product_created`
  - `product_updated`
  - `sku_deactivated`
- 支持多下游订阅
  - 搜索
  - 推荐
  - 审计
  - 风控
  - 数据分析
- 提供削峰和异步化能力
- 支持事件重放和历史追踪

### 20.4 Kafka 与 Saga 的关系

两者不是互斥关系。

更合理的演进方式是：

- 当前阶段：先用 `Saga` 解决商品和库存的一致性
- 后续阶段：在 Saga 之外，再用 Kafka 承接商品领域事件广播

也就是说：

- `Saga` 负责核心写链路的一致性
- Kafka 负责事务成功后的事件传播

不建议直接把 Kafka 当成当前事务问题的唯一解决方案，因为：

- 消息最终一致不能天然替代补偿语义
- 事件投递成功不等于业务事务已经定义完整

## 21. 当前结论

针对“商品创建成功但库存初始化失败”的问题，当前最合理的分布式事务方案是：

- 采用 `Saga`
- 使用 `DTM` 作为事务协调框架
- 由事务协调器统一编排 `product-service` 与 `inventory-service`
- 商品和库存都提供正向接口与补偿接口
- 通过事务分支日志保证幂等、空补偿、防悬挂
- 商品在事务完成前不应直接暴露为最终可售状态

Kafka 不应作为当前问题的前置依赖。

Kafka 更适合在后续系统进入事件驱动和多下游广播阶段时引入，承担领域事件总线职责，而不是替代当前的 Saga 事务协调。
