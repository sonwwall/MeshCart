# Questions

## 核心场景类

### 如何防止超卖？

当前项目已经落地了一版可用方案。

`inventory-service` 没有采用“先查库存再扣库存”的裸更新方式，而是把库存操作拆成：

1. `Reserve`
2. `Release`
3. `ConfirmDeduct`

具体来说：

- 下单时先预占库存
- 取消订单或创建订单失败时释放预占
- 支付成功后再确认扣减总库存

库存表更新也不是普通覆盖写，而是条件更新：

- 预占要求 `available_stock >= quantity`
- 确认扣减要求 `reserved_stock >= quantity`

这样至少能防止最直接的超卖。

项目对应实现可以看：

- [reservation.go](/Users/ruitong/GolandProjects/MeshCart/services/inventory-service/biz/service/reservation.go)
- [repository.go](/Users/ruitong/GolandProjects/MeshCart/services/inventory-service/biz/repository/repository.go)

如果后面出现热点 SKU 抢购，再继续补：

- 热点库存隔离
- 分片库存
- 队列化削峰

### 秒杀系统如何设计？

当前项目还没有秒杀专用链路。

如果基于现在的 MeshCart 往秒杀演进，我会按这四层设计：

1. 入口限流
2. 活动资格校验
3. 库存削峰
4. 订单异步化

也就是：

- Gateway 先挡住超过预算的流量
- 校验用户、活动、时间窗、购买次数
- 热点 SKU 不直接让所有请求同时打数据库
- 秒杀成功先返回“已受理/排队中”，再异步创建订单

对当前项目来说，如果真的开始做秒杀，第一步不是先写活动页，而是先补：

- Gateway Redis 分布式限流
- 热点库存削峰
- 订单异步受理机制

### 秒杀系统中如何做限流？

当前项目已经有第一阶段限流，但只是单机内存态。

也就是说现在适合：

- 单实例开发
- 基础防刷

但不适合真正的多实例高并发入口治理。

如果进入秒杀场景，限流必须前置到 Gateway，并且至少做三层：

1. 全局入口限流
2. 活动 / SKU 维度限流
3. 用户维度限流

真正进入多实例后，限流存储必须切 Redis 分布式实现。

### 秒杀中为什么要用 MQ？

当前项目主交易链路还没有使用 MQ。

但如果进入秒杀场景，MQ 的价值很明确：

- 削峰
- 排队
- 降低数据库和库存服务瞬时写压力
- 把同步强实时下单改成异步受理

所以 MQ 不是“为了高级而上”，而是当数据库和库存服务扛不住瞬时流量时，用来把流量拉平。

对当前项目来说，普通交易阶段还不需要先上 MQ。先做：

- 分布式限流
- 幂等强约束
- 热点观测

更值钱。

### 如何应对瞬时高并发流量？

结合当前项目，我建议按这个顺序做：

1. Gateway 限流
2. 写接口幂等
3. 下游超时预算
4. 热点读缓存
5. 必要时异步削峰

原因很简单：

- 先挡流量
- 再防重复执行
- 再保护下游
- 最后才考虑更重的异步架构

当前项目最该先做的仍然是：

- 把 `gateway` 从单机限流升级成 Redis 分布式限流

### 热点商品被大量访问怎么办？

当前项目还没有专门的缓存层。

如果热点商品访问量起来，最适合先缓存的是：

- 商品详情
- 商品列表 / 分类页结果
- SKU 基础映射信息

不建议优先缓存：

- 实时库存
- 订单状态
- 支付状态

因为这些字段对一致性的要求更高。

所以更准确的说法是：

- 热点商品信息应该缓存
- 热点库存信息不要先按普通页面缓存做

### 如何设计库存扣减逻辑？

当前项目已经采用三段式库存模型：

1. `Reserve`
2. `Release`
3. `ConfirmDeduct`

对应业务含义：

- 下单时先预占
- 取消订单或失败时释放
- 支付成功后真正扣减

这套模型的好处是：

- 下单成功但未支付时不会直接扣最终库存
- 订单过期关闭时可以释放预占
- 订单状态和库存状态更容易对齐

如果后面继续升级高并发治理，也应该围绕这套模型演进，而不是推翻重来。

### 订单和库存如何保证一致？

当前项目已经落地的是“同步编排 + 局部幂等 + 最终一致”。

具体是：

- 下单时先调用库存预占，再落订单
- 订单落库失败则释放库存
- 支付成功后，支付服务调订单支付确认
- 订单服务在状态迁移过程中再调用库存确认扣减
- 订单取消 / 超时关闭时调用库存释放预占

这说明当前项目不是靠一个大事务硬包住，而是靠：

- 顺序编排
- 幂等控制
- 失败补偿

来保证一致性。

### 如何保证接口幂等？

当前项目在 `order-service` 和 `payment-service` 里都已经有 action record 机制。

做法是：

- 为关键写动作生成 `(action_type, action_key)`
- 数据库里对这组键做唯一约束
- 同一动作重复进来时：
  - 已成功则返回历史结果
  - pending 则拦截
  - failed 则按具体语义处理

当前已经覆盖：

- 创建订单
- 取消订单
- 创建支付单
- 支付确认

但有一个边界：

- 这套机制依赖调用方提供稳定的 `request_id`
- 目前它还不是所有关键写接口的强约束

所以下一步应该把 `request_id` 从可选变成必填。

### 如何防止重复下单？

当前项目的答案是：

- 如果调用方传入稳定的 `request_id`
- `order-service` 会先查 action record
- 已成功的请求直接返回历史结果
- pending 中的请求直接拦截

这已经能处理大部分重复点击和请求重放问题。

但当前还差最后一步收口：

- 创建订单必须强制要求 `request_id`

不然没有幂等键时，重复点击仍然可能转成真实多次下单尝试。

### 如何处理订单支付失败但库存已扣的问题？

当前项目其实已经避开了“支付失败但库存已最终扣减”这个最糟糕的设计。

因为现在不是支付前就直接扣最终库存，而是：

- 下单先预占
- 支付成功后才确认扣减

所以这个问题被拆成两种：

1. 支付失败或支付过期
   - 库存只是预占，还没最终扣减
   - 后续通过订单关闭 / 取消释放库存
2. 支付成功后确认扣减失败
   - 当前会返回错误并阻断状态继续推进
   - 后面如果链路更复杂，再补异步补偿机制

### 如何设计一个高并发系统？

结合当前项目，我会把高并发系统设计拆成四层：

1. 入口治理
   - 限流
   - 超时
   - 熔断
2. 写链路治理
   - 幂等
   - 条件更新
   - 热点隔离
3. 读链路治理
   - 缓存
   - 降级
   - 只读重试
4. 异步化治理
   - 削峰
   - 延迟任务
   - 最终一致

对当前项目来说，最先要做的是：

- 分布式限流
- 幂等强约束

而不是一开始就把系统改成“全异步 + MQ + 分布式锁满天飞”。

## 架构设计类

### 电商系统如何拆分微服务？

结合当前项目，拆分方式已经比较清晰：

- `gateway` 负责统一鉴权、路由聚合、限流、用户入口
- `user-service` 负责账号、登录校验、角色信息
- `product-service` 负责商品、SKU、上下架
- `cart-service` 负责购物车
- `inventory-service` 负责库存初始化、可售校验、预占、释放、确认扣减
- `order-service` 负责订单创建、取消、关闭、支付确认
- `payment-service` 负责支付单创建、查询、关闭、mock 支付成功

这套拆分基本符合电商系统里“高内聚、低耦合、按领域拆分”的思路。

未来继续演进时，我不会急着再拆更多服务。下一步更值钱的是先把：

- 限流
- 幂等
- 缓存
- 异步任务

这些治理能力补强，而不是继续横向拆分。

### 如何设计订单系统？

当前项目的订单系统已经有一条完整主链路：

- 下单前做商品和库存校验
- 下单时先预占库存
- 订单落库失败则释放库存
- 订单创建后可取消、可超时关闭
- 支付成功后推进订单到 `paid`
- 订单支付确认时再触发库存确认扣减

这说明当前订单系统的设计重点不是“一个大事务”，而是：

- 订单状态机
- 幂等 action record
- 与库存、支付的顺序编排
- 异常时的补偿释放

如果后面继续升级，我建议增加：

- 强制 `request_id`
- 更清晰的超时关闭调度
- 订单状态日志查询能力
- 后续支付成功事件化

### 如何设计库存系统？

当前项目库存系统已经采用了比较稳妥的三段式模型：

1. `Reserve`
2. `Release`
3. `ConfirmDeduct`

这比“直接扣减库存”更适合电商交易链路，因为它能把“未支付订单”和“已支付订单”区分开。

当前已经做的重点包括：

- 预占时校验 `available_stock >= quantity`
- 确认扣减时校验 `reserved_stock >= quantity`
- 订单失败或取消时释放预占
- 商品创建时支持库存初始化 Saga

未来如果并发压力继续上来，再考虑：

- 热点 SKU 隔离
- 库存分片
- 基于队列的削峰消费

### 分布式事务有哪些解决方案？

常见方案主要有：

1. 本地事务 + 补偿
2. Saga
3. TCC
4. XA / 2PC
5. 事件驱动的最终一致性

当前项目实际已经用了两种：

- 商品创建链路：`gateway + DTM workflow + Saga`
- 订单交易链路：服务内同步编排 + 失败补偿 + 幂等收口

也就是说，这个项目不是所有地方都统一上一种重型事务模式，而是按场景选。

### TCC 和 SAGA 的区别是什么？

简单说：

- `TCC` 是 `Try / Confirm / Cancel`
- `Saga` 是正向动作 + 补偿动作

`TCC` 更强，但要求每个参与方都天然支持预留态和三段接口，开发成本更高。
`Saga` 更适合当前项目这种“先做业务，再在失败时回滚补偿”的场景。

当前项目实际选择的是 `Saga`，尤其是商品创建链路已经接入 `DTM workflow`。这个选择是合理的，因为：

- 商品创建和库存初始化更适合“失败后补偿删除”
- 当前系统还没有大规模的 Try/Confirm/Cancel 三段资源模型
- 直接上 `TCC` 成本高于收益

如果后面出现更复杂的资金预留、优惠券冻结、外部渠道预授权，再考虑局部引入 `TCC`。

### 如何实现最终一致性？

当前项目已经有一版实际落地：

- 下单先预占库存
- 订单落库失败释放库存
- 支付成功后订单再确认已支付
- 订单确认支付后库存再确认扣减
- 商品创建失败时通过 Saga 做补偿删除

这本质上就是：

- 不追求跨服务强一致大事务
- 通过状态机、幂等、补偿把系统收敛到最终一致

未来如果链路变得更长，比如再接入积分、优惠券、消息通知，再考虑：

- Outbox
- 领域事件
- 消息驱动补偿

### 服务之间如何通信？

当前项目主要是：

- 外部客户端到 `gateway`：HTTP
- `gateway` 到各服务：Kitex RPC
- 服务与服务之间：Kitex RPC

这套方式的优点是：

- 用户入口统一
- 服务间接口清晰
- 适合现在这种同步交易链路

后续如果增加更多异步场景，再引入 MQ 作为第二种通信方式，而不是替代 RPC。

### 如何做服务发现？

当前项目已经接入 `Consul` 做服务发现。

也就是说：

- 服务启动时注册到 Consul
- Gateway 和下游服务通过 Consul resolver 发现目标服务
- 多个服务还暴露了 `/healthz`、`/readyz`、`/metrics`

这对当前阶段已经够用。

未来如果规模继续变大，再考虑：

- 更严格的摘流顺序
- 多实例下的熔断和隔离
- 更细的服务治理平台

## 性能优化类

### 如何提升系统 QPS？

结合当前项目，提升 QPS 最有效的顺序不是盲目加机器，而是：

1. Gateway 限流
2. 写接口幂等
3. 热点读缓存
4. 下游超时和重试预算
5. 热点链路异步化

当前项目已经有：

- 第一阶段 Gateway 限流
- 关键写链路幂等骨架
- 统一的 RPC timeout
- 读 RPC 一次有限重试

但还没做：

- Redis 分布式限流
- 商品热点缓存
- 真正的熔断与隔离

所以当前最现实的提 QPS 手段，是先做这些，而不是一上来分库分表。

### 如何设计缓存架构？

当前项目还没有正式缓存层，所以这里更多是未来设计。

我建议优先缓存：

- 商品详情
- 商品列表 / 分类页
- SKU 基础只读信息
- 部分用户基础信息快照

不建议优先缓存：

- 实时库存
- 订单状态
- 支付状态

缓存层次建议是：

1. 先在 Gateway 或读服务前加 Redis 缓存
2. 短 TTL + 写后失效
3. 热点 key 再考虑本地缓存 + Redis 双层

第一版不要搞太复杂，先把商品详情和列表读流量挡住。

### 如何解决缓存穿透？

当前项目还没上正式缓存，但如果后面加缓存，穿透问题建议这样处理：

- 参数校验前置，非法 `product_id`、`sku_id` 直接拦截
- 对不存在的数据做空值缓存，TTL 短一些
- 热点接口前加限流，避免恶意刷随机 key

结合当前项目，最先会遇到穿透风险的是：

- 商品详情
- SKU 查询
- 用户查询

### 如何解决缓存击穿？

缓存击穿一般是某个热点 key 失效瞬间，大量请求同时打到下游。

当前项目还没做缓存，所以还没落这层保护。未来建议：

- 热点 key 提前续期
- 单飞控制，同一 key 只允许一个请求回源
- 给极热点商品加本地短缓存

对当前项目最典型的场景是热点商品详情页，而不是库存接口。

### 如何解决缓存雪崩？

缓存雪崩是大量 key 同时过期，导致流量一起打到数据库或下游服务。

未来设计时建议：

- TTL 加随机抖动
- 分批预热热点数据
- 缓存失效时配合限流和降级
- 保留兜底页面或默认响应

对当前项目来说，雪崩问题还不是眼前最急的，因为缓存层还没正式上。现在先做缓存架构本身更重要。

### 分库分表如何设计？

当前项目还没有到需要分库分表的阶段。

原因很简单：

- 服务数量已经拆开了
- 当前业务量还没有证据表明单表成为瓶颈
- 现在更大的问题在限流、幂等、缓存、热点治理

如果未来订单量起来，最可能先做分片的是：

- `orders`
- `order_items`
- `payments`

通常按这些维度选：

- `user_id`
- `order_id`
- 时间区间

但这一步要在真实数据量、慢 SQL、索引命中率都证明有必要后再做。

### 如何做读写分离？

当前项目还没有读写分离。

未来如果数据库读压力明显大于写压力，可以考虑：

- 主库负责写
- 从库负责列表、详情、后台查询等读流量

但交易核心链路要谨慎：

- 下单
- 支付确认
- 库存预占与确认

这些最好还是走主库或强一致读，不然会被复制延迟坑到。

所以对当前项目来说，读写分离优先级不如缓存。

### 如何优化数据库性能？

结合当前项目，数据库优化的顺序建议是：

1. 索引设计
2. SQL 访问路径
3. 减少无效重试
4. 热点缓存
5. 真正需要时再考虑分库分表

当前项目已经能看到一些正确方向：

- 订单 action record 有唯一索引
- 购物车 `(user_id, sku_id)` 有唯一约束
- 订单、库存、支付表都围绕核心查询路径建了索引

未来还应继续补：

- 慢 SQL 观测
- 热点行锁等待观测
- 商品和订单列表查询的分页压测

## 一致性与可靠性

### 如何保证接口幂等？

当前项目已经在 `order-service` 和 `payment-service` 做了 action record 幂等。

核心做法是：

- `(action_type, action_key)` 建唯一约束
- 同一请求如果已经成功，直接返回历史结果
- 如果还在处理中，直接拦截
- 如果之前失败，再按具体语义处理

它已经覆盖：

- 创建订单
- 取消订单
- 创建支付单
- 支付确认

真正还差的一步是把 `request_id` 从可选改成强制要求。

### 如何防止重复下单？

当前项目已经能防大部分重复下单，但前提是调用方传稳定的 `request_id`。

也就是说：

- 同一个 `request_id` 重放，不会真的多次创建订单
- 没有 `request_id` 时，这层保护就不完整

所以未来收口方向很明确：

- Gateway 对创建订单强制校验 `request_id`
- 后端继续保留数据库唯一约束和 action record

### 如何保证消息不丢失？

当前项目主交易链路还没有正式接入 MQ，所以这里属于未来设计。

如果后面开始用 MQ，我会这样做：

1. 业务数据先落本地库
2. 通过 Outbox 或可靠事件表再投递消息
3. 消费端做幂等
4. 投递失败可重试，必要时人工补偿

也就是说，不会走“业务写库和发消息各做各的”这种最容易丢消息的方案。

### MQ 如何保证消息不重复消费？

当前项目还没用 MQ，但未来设计一定要按“至少一次投递”来假设。

消费端保证不重复消费的方式通常是：

- 消费前先查幂等记录
- 使用业务唯一键去重
- 已处理过的消息直接返回成功

结合当前项目，最适合做消费幂等键的对象是：

- `order_id`
- `payment_id`
- `request_id`
- 领域事件 ID

不要指望 MQ 自己帮你解决所有重复消费问题，真正的幂等一定要落在业务侧。

### 如何处理订单支付失败但库存已扣的问题？

当前项目已经尽量避免这个问题：

- 下单时只是预占库存
- 不是直接扣最终库存
- 支付成功后才确认扣减

所以“支付失败但库存已扣”的主问题，已经被设计拆开了。

如果未来还有“支付成功后库存确认扣减失败”的长链路异常，再补：

- 异步补偿任务
- 对账任务
- 告警和人工修复入口

### 分布式锁如何实现？

当前项目没有把分布式锁作为主方案大面积使用。

这其实是对的，因为很多场景用：

- 数据库唯一约束
- 条件更新
- 幂等表

就已经足够，而且比分布式锁更稳。

如果未来确实需要分布式锁，通常会用 Redis：

- `SET key value NX PX ttl`
- 释放时校验 value 再删除

但我只会在“同一业务对象短时间内必须串行进入，且数据库约束不够表达”的场景里局部使用。

### Redis 分布式锁有什么问题？

主要问题有：

- 锁超时导致误释放
- 客户端暂停或 GC 导致锁过期
- 续约失败
- 时钟和网络抖动带来边界问题
- 把锁当作强一致事务工具会过度乐观

所以对当前项目来说，Redis 锁不应该优先拿来解决：

- 重复下单
- 支付幂等
- 库存扣减

这些问题更适合：

- 唯一约束
- 条件更新
- 幂等记录

Redis 锁更适合做一些管理任务串行化，而不是交易核心正确性的唯一防线。

## 中间件 & 技术细节

### Redis 在电商系统中的作用有哪些？

结合当前项目，Redis 已经开始承担登录会话管理：

- `refresh_token` 白名单
- `session_id` 到会话的映射
- 登出后的 session 吊销

未来在电商系统里，Redis 还适合承担：

- 分布式限流
- 商品热点缓存
- 短期幂等标记
- 排队令牌或秒杀资格控制

但不建议把它直接当作库存最终真相源。

### Redis 为什么快？

核心原因比较经典：

- 内存访问
- 单线程事件循环减少锁竞争
- 高效数据结构
- IO 多路复用

但要注意，Redis 快不代表什么都该放进去。像库存最终扣减、订单真相状态这类核心事实，仍然应该落数据库。

### Redis 如何实现限流？

当前项目第一阶段限流还不是 Redis，而是 Gateway 里的单机 token bucket。

如果后面升级成 Redis 分布式限流，我会这么做：

- 以 `route / user / ip / sku` 为 key
- 使用 Lua 保证计数和过期原子性
- 支持固定窗口、滑动窗口或令牌桶

对当前项目最先要迁移到 Redis 的接口是：

- 登录
- 注册
- refresh token
- 后台商品写接口

### MQ 为什么可以削峰？

因为 MQ 可以把“瞬时大量并发请求”变成“后端按固定速度消费”。

对当前项目来说，MQ 最可能未来用于：

- 热点下单削峰
- 支付成功后的异步事件扩散
- 超时关闭类延迟任务

但现在还没到必须上 MQ 的阶段。

### Kafka 和 RabbitMQ 有什么区别？

简单区分：

- `Kafka` 更偏高吞吐、日志流、事件流
- `RabbitMQ` 更偏业务队列、路由灵活、延迟和死信能力友好

如果以后 MeshCart 只是在交易系统里补：

- 延迟关闭订单
- 通知消息
- 支付成功事件

我会优先考虑 `RabbitMQ` 一类更贴业务队列的方案。

如果以后要做大规模事件流、埋点流、异步分析，再考虑 `Kafka`。

### Go 如何处理高并发？

Go 的主要优势是：

- goroutine 轻量
- channel 和 select 适合并发编排
- 网络服务模型简单直接

结合当前项目，Go 更适合做：

- 网关并发请求处理
- RPC 并发调用
- 后台异步任务和定时任务

但高并发不是“起更多 goroutine”就行，还是要配合：

- 限流
- 超时
- 幂等
- 资源隔离

### channel 有哪些使用场景？

常见场景包括：

- goroutine 间结果传递
- worker pool 任务分发
- 退出通知
- 超时控制配合

当前项目主交易链路还没有大量依赖 channel 做复杂并发编排，更多还是同步 RPC + 明确状态机，这对当前阶段是合理的。

如果以后加：

- 异步后台任务
- 本地 worker pool
- 热点预处理

channel 才会更常用。

### select 的作用是什么？

`select` 用来同时等待多个 channel 事件。

典型用途是：

- 等待结果或超时
- 多个任务谁先返回用谁
- 退出信号与业务处理并行监听

在当前项目里，如果后面做：

- 异步消费者
- 本地任务调度
- 聚合多个并发子任务

`select` 会比现在更常出现。

## 高可用与治理

### 什么是熔断？什么是降级？

熔断的重点是：

- 下游持续失败时，先快速拒绝，不再继续放大故障

降级的重点是：

- 在依赖不可用时，给一个更弱但还能工作的结果

对当前项目来说：

- 现在已经有超时、限流、基础错误码
- 但还没有真正系统化的熔断与降级

所以这部分属于下一阶段治理。

### 如何实现服务熔断？

当前项目还没正式上熔断。

未来实现顺序我建议是：

1. 先把超时和错误码口径统一
2. 再统计下游失败率、超时率
3. 对关键下游做失败快速返回

最适合先做熔断的不是核心写接口，而是高频读接口，比如：

- 商品详情
- 商品列表
- 用户信息只读查询

因为写接口盲目熔断很容易把业务语义弄坏。

### 如何实现服务降级？

当前项目还没有完整降级策略。

未来我会这样做：

- 商品详情下游抖动时，可返回缓存快照
- 非核心推荐、榜单、扩展信息可以直接降级为空
- 后台非关键统计接口可以直接提示稍后重试

但交易主链路：

- 下单
- 支付确认
- 库存预占

不适合做“伪成功式降级”，这里宁可失败返回，也不能返回错误成功。

### 如何做超时控制？

当前项目已经在这方面做了一版基础治理：

- Gateway 有 HTTP 总超时
- RPC client 显式配置 connect timeout 和 rpc timeout
- 文档里也已经明确后续要继续统一 DB / Redis / MQ 超时预算

下一步更重要的是继续把超时预算分层，而不是只配一个大超时：

- HTTP 总超时
- Gateway -> RPC 超时
- DB 超时
- Redis 超时

### 如何做重试？

当前项目的策略比较保守，这是对的。

现在已经做的是：

- 只给幂等读 RPC 配一次有限重试
- 写 RPC 默认不自动重试

这点在代码里已经体现了，`NewReadFailureRetryPolicy` 默认最多只重试一次。

后续如果继续做重试治理，我会坚持：

- 只有真正幂等的读操作允许自动重试
- 写接口不要默认自动重试
- 重试一定受超时预算约束

### 什么是服务雪崩？如何解决？

服务雪崩就是一个下游依赖变慢或不可用后，上游不断堆积请求、超时、重试，最终把更多服务一起拖垮。

对当前项目来说，最容易触发雪崩的位置通常是：

- Gateway 到商品、用户、库存等高频 RPC
- 高峰期热点读接口

解决顺序一般是：

1. 超时
2. 限流
3. 重试收敛
4. 熔断
5. 降级
6. 隔离

### 什么是服务隔离？

服务隔离就是把不同流量、不同依赖、不同资源池分开，避免一个热点把全站拖死。

对当前项目未来可以做的隔离包括：

- 核心链路和后台链路隔离
- 慢接口和快接口隔离
- 热点 SKU 相关流量隔离
- 不同下游 client 的连接池和并发预算隔离

当前项目还没做到这一步，但如果高并发治理继续推进，这会是限流和熔断后的下一层。

## 灵魂拷问

### 如果 Redis 挂了怎么办？

结合当前项目，Redis 目前主要承载的是登录会话白名单，还没有成为交易主链路的硬依赖。

所以 Redis 挂掉的影响主要是：

- refresh token 无法校验
- logout 无法吊销 session
- 如果以后接入分布式限流和缓存，相关能力也会受影响

我会建议的处理方式是：

- 登录态相关接口直接失败，不伪成功
- `/me` 这类只验 access token 的接口仍可用
- 把 Redis 故障影响面控制在 auth/cache/rate-limit，而不是拖垮交易主链路

### 如果数据库挂了怎么办？

数据库挂掉时，交易核心服务基本都无法正常工作。

这时真正正确的做法不是“想办法继续下单”，而是：

- 快速失败
- 保护线程和连接池
- 避免重试风暴
- 保证系统有序降级

对当前项目来说，数据库仍然是订单、支付、库存、商品的真相源，所以 DB 故障是最高优先级故障。

### 如果 MQ 挂了怎么办？

当前项目还没正式接 MQ，所以现在这个问题还不是生产现实问题。

未来如果接入 MQ，我会这样设计：

- MQ 只承接异步链路，不直接成为同步主交易唯一依赖
- 投递失败可重试
- 消费失败可重试和死信
- 核心真相仍在数据库

也就是说，MQ 挂了应该影响异步扩展能力，不应该直接让数据库事实丢失。

### 如何保证系统不崩？

不能靠某一个组件保证，而是靠一整套治理。

结合当前项目，顺序应该是：

1. 参数校验
2. 超时
3. 限流
4. 幂等
5. 失败补偿
6. 日志、metrics、trace
7. 熔断、降级、隔离

当前项目已经有前半段基础，真正还缺的是第二阶段治理能力。

### 系统瓶颈一般在哪里？

结合当前项目，未来最可能先出现瓶颈的地方是：

1. Gateway 多实例下的限流失效
2. 热点商品详情读流量
3. 热点 SKU 的库存行锁竞争
4. 订单和支付写热点
5. 下游 RPC 放大和超时堆积

也就是说，当前项目的第一批瓶颈更可能出现在：

- 入口治理
- 热点读
- 库存写热点

而不是一开始就出在“分库分表没做”。

### 如何做系统压测？

对当前项目，压测不能一上来全链路乱打，建议分三层：

1. 单接口压测
   - 登录
   - 商品详情
   - 下单
   - 创建支付单
2. 交易链路压测
   - 商品 -> 加购 -> 下单 -> 支付成功
3. 热点场景压测
   - 单 SKU 热点访问
   - 单 SKU 高频下单

压测时要同时看：

- RT
- 错误率
- DB CPU / 锁等待
- Redis RT
- RPC timeout
- 限流命中率

对当前项目来说，先做单接口和热点 SKU 压测，比直接做超大规模全链路压测更有价值。
