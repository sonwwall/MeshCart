# MeshCart 压测瓶颈分析

本文档基于 `2026-03-24` 已完成的 7 轮 `k6` 压测结果整理，目标不是重复记录每一轮参数，而是沉淀当前阶段可以成立的性能判断、瓶颈假设和后续验证方向。

详细执行过程、原始结果文件和逐轮记录见 [k6-loadtest-plan.md](/Users/ruitong/GolandProjects/MeshCart/docs/k6-loadtest-plan.md)。

## 1. 当前总体判断

截至第七轮，系统呈现出以下总体特征：

1. 读链路整体稳定，当前没有暴露出明显的吞吐瓶颈
2. 默认网关限流会先于应用本身成为压测干扰项
3. 交易写链路中，`create order` 明显比其他接口更敏感
4. 订单链路的异常放大暂时没有稳定复现为“固定单点瓶颈”
5. 测试数据状态，尤其是热点 SKU 库存，已经成为影响压测结论的重要变量

换句话说，当前系统的问题不是“随便一压就崩”，而是“写链路更脆弱，且结论容易被库存状态和环境波动污染”。

## 2. 已经比较确定的结论

### 2.1 读链路不是当前主要瓶颈

从第一到第三轮结果看：

- `GET /api/v1/products`
- `GET /api/v1/products/detail/:product_id`

在接近 `100 req/s` 的量级下仍然保持了比较低的 `p95`，没有出现明显错误放大，也没有出现类似订单写链路那种阶跃式 RT 抬升。

这说明当前阶段：

- `gateway`
- `product-service`

组成的主读路径整体比较稳，不是当前最优先的性能风险点。

### 2.2 默认限流是第一层“假瓶颈”

第二轮一开始，最先碰到的问题不是服务本身过载，而是网关限流提前生效：

- 登录在默认阈值附近开始被拒绝
- 商品列表升压时也明显受到限流影响

这意味着：

1. 从生产保护角度看，限流配置是有效的
2. 从性能分析角度看，默认限流会污染压测结果

因此如果要继续做性能压测，必须先区分两类问题：

- 是系统真的慢了
- 还是护栏先把流量拦掉了

### 2.3 登录链路重于商品读链路，但尚未成为核心瓶颈

登录接口的 RT 长期高于商品读接口，但在当前测试强度下没有表现出明显的不稳定性，也没有出现写链路那种大幅波动。

因此现阶段更适合把登录链路定义为：

- 成本较高的入口接口

而不是：

- 当前系统最优先的性能瓶颈

### 2.4 支付链路有抬升，但优先级低于订单链路

第三、四轮里，`POST /api/v1/payments` 的耗时有过明显上升，但整体趋势仍然比 `POST /api/v1/orders` 更平滑，也没有在后续轮次中反复被证明是最先失稳的点。

因此目前支付链路的判断是：

- 值得持续观察
- 但不是当前最急的瓶颈

## 3. 当前最值得关注的链路

### 3.1 订单创建链路是当前最敏感的写路径

第四轮和第五轮中，`POST /api/v1/orders` 在 `20 VUs` 下曾出现明显 RT 放大：

- 第四轮 `p95` 接近 `700ms`
- 第五轮 `p95` 达到 `1.03s`

这说明订单链路具备“被放大”的条件，是当前最值得优先分析的写路径。

虽然第六、第七轮没有稳定复现同等级的高延迟，但这并不推翻订单链路的风险，只能说明：

- 它是敏感点
- 但根因还没有被稳定锁定

### 3.2 gateway 不是主要耗时来源，更像是在等待下游

第五轮做了指标前后对比后可以看到：

- `gateway /api/v1/orders` 平均耗时增量
- `order-service create_order` 平均耗时增量

两者几乎同步。

这说明 `gateway` 本身并没有表现出独立的耗时膨胀，更合理的解释是：

- gateway 主要是在等待订单 RPC 和下游链路完成

所以当前不应优先怀疑：

- 网关本地 JSON 处理
- 网关 handler 逻辑
- 网关本地 CPU

更应该优先看：

- `order-service`
- `inventory-service`
- 以及它们背后的数据库操作

### 3.3 库存预占是当前最像“热点放大器”的一段

第五轮指标中，库存服务的 `reserve_sku_stocks` 耗时增量明显高于产品服务相关读 RPC。

这意味着在订单编排链路里：

- 商品读取
- SKU 查询

虽然有成本，但目前还没有库存预占那么值得警惕。

现阶段最值得优先关注的调用路径是：

`gateway -> order-service -> inventory-service reserve_sku_stocks -> MySQL`

这个结论是目前最接近“可操作瓶颈判断”的部分。

## 4. 当前不能直接下结论的地方

### 4.1 还不能直接认定“数据库行锁”就是根因

虽然库存预占很可疑，但当前证据仍然缺一层：

- MySQL 锁等待
- 慢查询
- 活跃事务
- InnoDB 行锁细节

由于当前环境没有直接可用的 `mysql` 客户端，这部分证据尚未补齐。

因此目前只能说：

- 数据库相关操作很值得怀疑

不能直接说：

- 订单抖动已经确定是数据库锁竞争导致

### 4.2 还不能证明“热点 SKU 竞争必然导致高 RT”

第六轮专门做了热点 SKU 和普通 SKU 对照。

补库存后的结果显示：

- 热点 SKU 和普通 SKU 的 `avg`
- `p95`
- 关键 RPC 平均耗时

都非常接近。

因此当前没有足够证据支持下面这个命题：

- “热点 SKU 自身就是订单链路高延迟的决定性原因”

更合理的说法是：

- 热点 SKU 是需要控制的变量
- 但不是目前已经被证明的唯一根因

### 4.3 第五轮高延迟更像“阶段性波动”，不是稳定极限

第七轮连续复测后，在补足库存的前提下，`create order 20 VUs` 的有效样本 `p95` 分别为：

- `329.14ms`
- `142.64ms`
- `137.82ms`

这和第五轮 `1.03s` 的结果差距很大。

因此目前最合理的判断是：

- 第五轮高延迟是真实观测
- 但它不像是当前环境下一按就能稳定复现的固定瓶颈

更像是：

- 某次环境波动
- 某段链路瞬时抖动
- 或测试数据状态叠加导致的放大结果

## 5. 当前真正暴露出来的流程问题

### 5.1 测试数据状态已经影响压测结论

第六轮普通 SKU 对照、第七轮热点 SKU 复测，都出现了同一种问题：

- 样本中后段被库存耗尽污染

这带来两个直接后果：

1. 压测结果会混入“库存不足”业务失败
2. 很容易把数据样本失效误判成系统性能问题

所以当前压测流程里最应该先补的一环不是更复杂的脚本，而是：

- 压测前库存检查
- 自动补库存
- 每轮结束后的库存回收或重置策略

### 5.2 当前系统对“连续多轮压测”敏感

从结果看，单轮压测本身问题不大，但当多轮连续针对同一热点 SKU 反复施压时：

- reserved stock 持续累积
- saleable stock 最终归零

说明当前环境下测试数据不会自动恢复到“干净初始态”。

这不是生产故障，但它是压测流程上的一个真实风险。

## 6. 现阶段最合理的瓶颈画像

结合 7 轮结果，当前系统的瓶颈画像更接近下面这个描述：

- 读链路稳
- 登录链路偏重但可控
- 支付链路有抬升趋势但不是当前主问题
- 订单写链路最敏感
- 订单写链路里最值得怀疑的是库存预占及其数据库依赖
- 但“稳定单点瓶颈”还没有被完整锁死
- 测试数据状态已经成为不可忽略的干扰变量

如果要用一句话概括：

当前系统的主要性能风险在交易写链路，尤其是订单创建时对库存预占的依赖；但当前阶段更像是“易受状态和波动影响的敏感链路”，而不是已经被证明的固定硬瓶颈。

## 7. 下一步建议

### 7.1 优先补压测流程，而不是盲目继续升压

建议先补这两项：

1. 压测前库存检查
2. 自动补库存或数据重置

否则后续结果会反复被样本失效污染。

### 7.2 再补数据库层证据

继续分析订单链路前，建议补齐：

- MySQL 锁等待
- MySQL 慢查询
- InnoDB 事务状态
- 订单表、库存表相关 SQL 耗时

只有拿到这层证据，才能更有把握判断：

- 是事务争用
- 是行锁竞争
- 是某类 SQL 本身慢

### 7.3 复测策略建议

如果继续后续轮次，建议优先采用下面的验证顺序：

1. 固定 `create order 20 VUs`
2. 每轮前确认热点 SKU 可售库存
3. 每轮只改一个变量
4. 同步抓 `gateway`、`order-service`、`inventory-service` 指标
5. 补数据库观测后，再判断是否继续升压

## 8. 当前结论的可信度边界

当前文档中的结论可以分成两类。

可信度较高的：

- 读链路不是当前主要瓶颈
- 默认限流会显著污染压测结果
- 订单创建链路比支付链路更敏感
- 库存预占是当前最值得观察的一段
- 测试数据状态会显著影响结论

仍需补证据的：

- 数据库是否是主要根因
- 是否存在稳定可复现的热点争用瓶颈
- 第五轮高延迟的精确触发条件

因此，当前最稳妥的技术判断不是“已经找到唯一瓶颈”，而是：

已经把问题收敛到订单创建和库存预占这一小段链路，下一步需要数据库层证据来完成定性。
