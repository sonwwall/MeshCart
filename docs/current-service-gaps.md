# 当前已完成部分问题清单

## 1. 目的

本文档用于记录当前已经落地的 `gateway / product-service / inventory-service / cart-service / order-service / payment-service` 主链路中，仍然存在的实现缺口与协作问题。

本文档只记录“已经确认存在、需要继续开发解决”的问题，不重复描述已完成的设计方案。

## 2. 优先级说明

- `P0`：会直接造成核心业务数据错误、明显脏数据，或让主链路对外表现和真实落库状态不一致
- `P1`：不会立刻阻断主链路，但会造成绕过校验、数据不一致、后台运维困难
- `P2`：体验、治理或可维护性问题，建议后续收口

## 3. 问题清单

### 3.1 P1：购物车更新数量仍缺少库存校验

当前现状：

- 购物车加购链路已经在进入 `cart-service` 前做商品状态和库存校验
- 但购物车更新数量链路还没有完全复用相同的库存校验口径

影响：

- 用户可能先加购成功，再通过更新数量把购物车项改到超过当前可售库存
- 这会让购物车视图和真实可售库存之间出现偏差

建议方案：

- 在购物车更新数量链路补齐与加购一致的商品状态校验
- 同步调用 `inventory-service.CheckSaleableStock`
- 保持“加购”和“改数量”两条入口使用同一套校验规则

### 3.2 P2：交易主链路仍缺少端到端联调回归

当前现状：

- `product-service`、`inventory-service`、`order-service`、`payment-service` 均已实现核心 RPC 和状态流转
- 仓库内也已有较完整的单测与部分验收测试
- 但“商品创建/上架 -> 加购 -> 下单 -> 创建支付单 -> 支付成功 -> 库存确认扣减”还没有沉淀成稳定的端到端回归入口

影响：

- 单服务测试通过，不代表跨服务编排一定稳定
- 后续变更容易在订单、支付、库存交界处引入回归

建议方案：

- 补齐面向交易主链路的验收测试
- 覆盖订单过期、支付重复确认、库存释放与确认扣减等关键分支
- 把这条链路作为后续 CI 的重点回归对象

### 3.3 P2：库存归属校验依赖商品详情逐个查询，后续会有性能放大问题

当前现状：

- 后台库存查询 / 调整时，`gateway` 先批量查 SKU
- 再按 `spu_id` 逐个调用商品详情查询创建者

影响：

- 单次批量库存查询会放大成多次 RPC
- SKU 数量和 SPU 数量变多后，后台接口性能会抖动

建议方案：

- 后续在商品服务补充更适合库存权限校验的批量接口
- 例如按 `sku_id` 直接返回 `sku_id -> spu_id -> creator_id`
- 避免后台库存接口对商品详情做 N 次调用

---

## 4. 当前结论

此前的 `P0`“商品创建失败但已落库的补偿收口”已经完成：

- `gateway` 已通过 `DTM workflow` 编排商品创建和库存初始化
- `product-service` / `inventory-service` 已实现 Saga 正向分支与补偿分支
- 库存初始化失败时，商品侧会执行补偿回滚

当前建议优先处理的是一个 `P1`：

1. 购物车更新数量补库存校验

随后在 `P2` 阶段收口：

1. 交易主链路端到端联调回归
2. 优化库存归属校验的批量查询方式
