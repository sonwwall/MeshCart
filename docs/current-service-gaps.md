# 当前已完成部分问题清单

## 1. 目的

本文档用于记录当前已经落地的 `gateway / product-service / inventory-service / cart-service` 主链路中，仍然存在的实现缺口与协作问题。

本文档只记录“已经确认存在、需要继续开发解决”的问题，不重复描述已完成的设计方案。

## 2. 优先级说明

- `P0`：会直接造成核心业务数据错误、明显脏数据，或让主链路对外表现和真实落库状态不一致
- `P1`：不会立刻阻断主链路，但会造成绕过校验、数据不一致、后台运维困难
- `P2`：体验、治理或可维护性问题，建议后续收口

## 3. 问题清单

### 3.1 P0：商品更新新增 SKU 时没有初始化库存

当前现状：

- `gateway` 在创建商品时会调用 `inventory-service.InitSkuStocks`
- 但更新商品时，如果请求里新增了一个 SKU，只会写入 `product-service`
- 新增 SKU 不会同步创建库存记录

影响：

- 商品侧存在新 SKU，但库存侧没有对应库存记录
- 后台库存查询、调整时会出现“SKU 存在但库存不存在”
- 前台加购该 SKU 时会在库存校验阶段失败

已确定方案：

- 更新商品时，如果请求里新增 SKU，必须支持同时传入 `initial_stock`
- `gateway` 在 `UpdateProduct` 成功后，识别本次新增的 SKU
- 对新增 SKU 调用 `inventory-service.InitSkuStocks`
- 未传 `initial_stock` 时，默认按 `0` 初始化，与创建商品保持一致

建议处理顺序：

1. 扩展更新商品请求中的 SKU 输入，允许新增 SKU 携带 `initial_stock`
2. 在 `gateway` 识别“新增 SKU 集合”
3. 在商品更新成功后，仅对新增 SKU 初始化库存
4. 为这条链路补单测和联调测试

---

### 3.2 P0：商品创建在库存初始化失败时会留下“接口失败但商品已创建”的脏数据

当前现状：

- `gateway` 先调用 `product-service.CreateProduct`
- 再调用 `inventory-service.InitSkuStocks`
- 如果库存初始化失败，接口会返回失败
- 但商品和 SKU 实际已经落库
- 当前只会对原本请求为 `online` 的商品做“最佳努力下架”

影响：

- 调用方认为创建失败，但库里实际上已经有商品数据
- 管理端重复重试会制造重复商品
- 会产生“接口语义失败”和“数据实际成功”不一致的问题

当前建议：

- 这是当前阶段允许存在的已知缺口，但必须在文档和后续计划中明确
- 短期继续保留“最佳努力下架”兜底
- 后续需要补更严格的一致性方案

后续方向：

- 异步补偿任务
- 后台失败重试入口
- 更严格的分布式一致性方案

---

### 3.3 P1：商品删除 SKU 时没有同步冻结库存

当前现状：

- 商品更新时，如果请求中删除了某个 SKU，`product-service` 会删除商品侧 SKU 记录
- 库存侧不会同步删除，也不会冻结

影响：

- 会形成孤儿库存记录
- 后台库存查询仍能看到已被商品侧删除的 SKU 库存
- 后续订单预占、库存流水、审计时边界会混乱

已确定方案：

- 不做默认物理删除库存记录
- 采用“商品删除 / SKU 删除 -> 库存冻结”的方案
- 库存记录保留，但状态收口为不可售

建议处理顺序：

1. 给库存域补状态字段或冻结能力
2. 商品更新删除 SKU 时，由 `gateway` 同步编排冻结库存
3. 后台库存查询接口补充库存状态展示

---

### 3.4 P1：购物车更新数量时没有重新做商品与库存校验

当前现状：

- 加购链路已经做了商品在线、SKU active、库存足够的校验
- 但购物车更新数量时，`gateway` 直接调用 `cart-service.UpdateCartItem`
- 没有重新检查商品状态和库存

影响：

- 用户可以先加购成功，再把数量改到超过库存
- 商品下架后，购物车项仍可能被继续修改
- 不一致问题会被延后到更后面的确认单或下单阶段

建议方案：

- 在购物车更新数量链路中补齐与加购一致的商品、SKU、库存校验
- 至少在数量上调时重新做库存校验

---

### 3.5 P2：库存归属校验依赖商品详情逐个查询，后续会有性能放大问题

当前现状：

- 后台库存查询 / 调整时，`gateway` 先批量查 SKU
- 再按 `spu_id` 逐个调用商品详情查询创建者

影响：

- 单次批量库存查询会放大成多次 RPC
- SKU 数量和 SPU 数量变多后，后台接口性能会抖动

建议方案：

- 后续在商品服务补充更适合库存权限校验的批量接口
- 例如按 `sku_id` 直接返回 `sku_id -> spu_id -> creator_id`
- 避免后台库存接口对商品详情做 N 次调用

## 4. 当前结论

当前最优先处理的是两个 `P0`：

1. 更新商品新增 SKU 时，支持携带 `initial_stock`，未传时默认按 `0` 初始化，并同步初始化库存
2. 明确商品创建失败但已落库的补偿策略，并在后续阶段收口

随后处理两个 `P1`：

1. SKU 删除改为库存冻结
2. 购物车更新数量补库存校验

`P2` 可以放在后台治理阶段一并优化。
