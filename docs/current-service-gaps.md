# 当前已完成部分问题清单

## 1. 目的

本文档用于记录当前已经落地的 `gateway / product-service / inventory-service / cart-service` 主链路中，仍然存在的实现缺口与协作问题。

本文档只记录“已经确认存在、需要继续开发解决”的问题，不重复描述已完成的设计方案。

## 2. 优先级说明

- `P0`：会直接造成核心业务数据错误、明显脏数据，或让主链路对外表现和真实落库状态不一致
- `P1`：不会立刻阻断主链路，但会造成绕过校验、数据不一致、后台运维困难
- `P2`：体验、治理或可维护性问题，建议后续收口

## 3. 问题清单

### 3.1 P0：商品创建在库存初始化失败时会留下“接口失败但商品已创建”的脏数据

当前现状：

- `gateway` 先调用 `product-service.CreateProduct`
- 再调用 `inventory-service.InitSkuStocks`
- 如果库存初始化失败，接口会返回失败
- 但商品和 SKU 实际已经落库
- 当前只会对原本请求为 `online` 的商品做“最佳努力下架”

影响：

- 调用方认为创建失败，但库里实际上已经有商品数据
- 管理端重复重试会制造重复商品
- 会产生“接口语义失败”和“数据实际成功”不一致的问题

当前建议：

- 这是当前阶段允许存在的已知缺口，但必须在文档和后续计划中明确
- 短期继续保留“最佳努力下架”兜底
- 后续需要补更严格的一致性方案

后续方向：

- 异步补偿任务
- 后台失败重试入口
- 更严格的分布式一致性方案

---

### 3.2 P1：商品删除 SKU 时没有同步收口商品与库存状态

当前现状：

- 商品更新时，如果请求中删除了某个 SKU，`product-service` 会删除商品侧 SKU 记录
- 库存侧不会同步删除，也不会冻结

影响：

- 商品侧删除 SKU 后，历史订单、审计、排障缺少商品侧参照
- 库存侧会形成语义不完整的孤儿库存记录
- 后台库存查询仍能看到已被商品侧删除的 SKU 库存
- 后续订单预占、库存流水、审计时边界会混乱

已确定方案：

- 不做默认物理删除商品 SKU 记录
- 不做默认物理删除库存记录
- 采用“商品 SKU 失效 + 库存冻结”的方案
- 商品侧保留 SKU 记录，但状态收口为不可售
- 库存侧保留库存记录，但状态收口为 `frozen`

建议处理顺序：

1. 给商品 SKU 补更清晰的失效状态语义，或至少约定删除动作改为 `inactive`
2. 给库存域补状态字段或冻结能力
3. 商品更新删除 SKU 时，由 `gateway` 同步编排“SKU 失效 + 库存冻结”
4. 后台商品详情和库存查询接口补充状态展示

---

### 3.3 P1：购物车更新数量时没有重新做商品与库存校验

当前现状：

- 加购链路已经做了商品在线、SKU active、库存足够的校验
- 但购物车更新数量时，`gateway` 直接调用 `cart-service.UpdateCartItem`
- 没有重新检查商品状态和库存

影响：

- 用户可以先加购成功，再把数量改到超过库存
- 商品下架后，购物车项仍可能被继续修改
- 不一致问题会被延后到更后面的确认单或下单阶段

建议方案：

- 在购物车更新数量链路中补齐与加购一致的商品、SKU、库存校验
- 至少在数量上调时重新做库存校验

---

### 3.4 P2：库存归属校验依赖商品详情逐个查询，后续会有性能放大问题

当前现状：

- 后台库存查询 / 调整时，`gateway` 先批量查 SKU
- 再按 `spu_id` 逐个调用商品详情查询创建者

影响：

- 单次批量库存查询会放大成多次 RPC
- SKU 数量和 SPU 数量变多后，后台接口性能会抖动

建议方案：

- 后续在商品服务补充更适合库存权限校验的批量接口
- 例如按 `sku_id` 直接返回 `sku_id -> spu_id -> creator_id`
- 避免后台库存接口对商品详情做 N 次调用

---

### 3.5 P2：缺少管理员商品聚合详情接口

当前现状：

- 当前有前台商品详情接口
- 当前也有后台商品列表和后台库存查询接口
- 但没有“管理员查看单个商品完整详情”的聚合接口

影响：

- 管理员排查单个商品时，需要分别调用商品详情和库存接口自行拼装
- 后台页面如果要展示 SKU 维度库存、商品状态、创建者信息，会增加前端复杂度
- 权限校验和聚合逻辑分散，不利于后续继续扩展后台视图

建议方案：

- 新增管理员商品详情接口，由 `gateway` 聚合商品服务和库存服务
- 接口建议形态为：`GET /api/v1/admin/products/:product_id`
- 返回内容除商品和 SKU 基础信息外，还应包含每个 SKU 的库存信息
- `admin` 仅可查看自己创建的商品，`superadmin` 可查看全量商品

## 4. 当前结论

当前最优先处理的是一个 `P0`：

1. 明确商品创建失败但已落库的补偿策略，并在后续阶段收口

随后处理两个 `P1`：

1. SKU 删除改为商品 SKU 失效 + 库存冻结
2. 购物车更新数量补库存校验

`P2` 可以放在后台治理阶段一并优化，包括：

1. 优化库存归属校验的批量查询方式
2. 补管理员商品聚合详情接口
