# MeshCart 基于 MQ 的库存削峰与订单异步化推进计划

本文档不再以“小机器单机可跑”为约束，而是假设 MeshCart 部署在较大规模集群环境中，目标是支撑更高并发、热点商品抢购、以及更强的业务扩展能力。

但需要先明确本文档的定位：

1. 它是 MeshCart 在当前稳定基线上的“下一阶段演进方案”
2. 它不是“系统已经扛不住，所以必须立刻全面 MQ 化”的结论
3. 它也不是一套完整的秒杀专用架构

文档重点回答 4 个问题：

1. 为什么当前阶段要考虑 `MQ`
2. 库存削峰应该如何基于 `MQ` 演进
3. 订单异步化应该如何分阶段推进
4. 这套方案如果落地，如何在简历中体现工程价值

原有同步链路优化与压测结论见：

- [k6-loadtest-plan.md](/Users/ruitong/GolandProjects/MeshCart/docs/k6-loadtest-plan.md)
- [high-concurrency-optimization-priorities.md](/Users/ruitong/GolandProjects/MeshCart/docs/high-concurrency-optimization-priorities.md)

## 1. 背景与目标

基于前 20 多轮压测，MeshCart 已经得到一版可用的稳定基线：

1. 商品 / SKU 读取批量化和 Redis 缓存已经落地
2. 订单幂等路径已经明显收缩
3. 订单主链路已经可以稳定走通
4. 热点库存的“稳定优先”实现已经能支撑更高档位热点压测

基于当前压测结论，更合理的判断不是“现在必须立刻把主交易链路 MQ 化”，而是：

1. 当前同步主链路已经形成一版可用稳定基线
2. 后续可以开始补异步事件基础设施，为下一阶段演进做准备
3. 如果未来部署形态升级到多实例，或者热点 SKU 竞争继续放大，现有本地热点治理会逐步接近边界

但如果目标继续升级到“更高并发、大促、热点 SKU 极端集中流量”，仅靠同步 RPC + 数据库事务，最终仍会遇到两个上限：

1. 热点库存写竞争上限
2. 主链路同步步骤过多导致的时延和容量上限

因此，下一阶段值得引入 `MQ` 的场景，不是替代整个交易闭环，而是：

1. 用于热点库存削峰
2. 用于订单后置动作异步化
3. 在必要时，用于热点订单的局部异步受理

### 1.1 当前阶段的不做项

为了避免方案在落地时失焦，先明确当前阶段不做什么：

1. 不把普通订单创建整体改成纯异步
2. 不把支付成功确认这类强一致主链路直接改成纯消息驱动
3. 不把所有 SKU 一次性切到库存排队模型
4. 不把强一致库存判断改成缓存优先

## 2. 设计原则

这套方案必须坚持下面几个原则。

### 2.1 核心交易结果优先保证一致性

以下动作仍然要被视为核心交易步骤：

1. 商品 / SKU 基础校验
2. 热点资格校验
3. 库存可用性判断
4. 订单主记录创建
5. 支付成功后的订单状态确认

这些步骤不能简单因为引入 `MQ` 就完全异步化，否则会让一致性、幂等、用户体验急剧复杂化。

### 2.2 非核心后置动作优先异步化

更适合通过 `MQ` 处理的是：

1. 站内消息
2. 短信 / 邮件通知
3. 审计日志
4. 统计埋点
5. 营销触发
6. 积分变更
7. 优惠券发放 / 核销后处理

### 2.3 只对热点场景引入更重的异步模型

不是所有订单都要改成排队异步下单。

更合理的策略是：

1. 普通订单继续走同步主链路
2. 热点 SKU 命中时，切入排队或局部异步受理
3. 用热点治理隔离极端流量，不污染普通交易体验

### 2.4 所有异步链路都必须建立在幂等与可观测之上

任何 `MQ` 方案落地前，都必须补齐：

1. 稳定的业务幂等键
2. 可追踪的订单状态机
3. 失败重试与死信处理
4. 完整的消息投递 / 消费指标

### 2.5 推进顺序优先于方案完整度

本文档里的几个方案，不代表要并行开工。

更合理的理解应该是：

1. 先补 `outbox + MQ` 的事件基础设施
2. 再把非核心后置动作事件化
3. 最后再评估是否要做热点 SKU 的局部排队
4. 热点订单局部异步受理只作为更后续的预备选项

### 2.6 当前已完成到哪里

截至当前版本，`P0` 基础设施已经落地了第一版骨架，但还没有开始写入真实业务事件。

已经完成的部分：

1. `docker-compose` 已补 Kafka，本地环境可以和 MySQL、Redis 一起启动
2. `app/mq` 已补共享基础件：
   - 通用消息 `envelope`
   - 通用 `outbox` 状态定义
   - 通用后台投递器 `dispatcher`
   - 基于 Gorm 的通用 `outbox` store
3. `order-service` 已增加 `order_outbox` 表模型和数据库迁移
4. `payment-service` 已增加 `payment_outbox` 表模型和数据库迁移
5. 新增基础件和订单、支付服务现有测试已通过，没有破坏当前主链路

当前还没有做的部分：

1. 业务事务内真实写入 `outbox`
2. Kafka producer 真正投递
3. 第一批领域事件发布
4. 下游消费者与死信处理闭环

## 3. 库存削峰的 MQ 化推进方案

## 3.1 当前库存削峰现状

当前项目已经有第一层热点治理：

1. `inventory-service` 按 `sku_id` 做本地限并发
2. 预占流程使用“先更新库存，再写 reservation”的稳定顺序
3. 通过压测验证，这版实现已经能稳定支撑 `hot-v80`

这说明当前版本已经不是“完全没有削峰”，但它的边界也很明显：

1. 本地限并发只在单实例有效
2. 多实例部署后，同一热点 `sku` 仍可能同时打到数据库
3. 它更适合当前稳定基线，不是最终的大规模热点方案

## 3.2 下一阶段目标

库存削峰的下一阶段，不是直接上库存分片，而是先评估“按热点 SKU 的 MQ 排队化”。

目标是：

1. 把同一热点 `sku` 的瞬时并发写，收敛成受控消费速率
2. 避免大量请求同时竞争同一库存行
3. 把数据库竞争问题前移到队列层面消化

## 3.3 推荐架构

如果后续压测和部署形态都表明本地热点治理已接近边界，可以增加一个“热点库存队列层”，总体链路如下：

1. `gateway` / `order-service` 识别请求是否命中热点 SKU
2. 非热点 SKU 继续走同步库存预占
3. 热点 SKU 请求写入 `MQ`
4. `inventory-worker` 按 `sku_id` 分区顺序消费
5. 消费端执行库存预占
6. 预占结果回写订单状态或通知上游

### 关键设计点

#### 分区策略

消息必须按热点维度有序处理，常见做法：

1. 以 `sku_id` 作为 partition key
2. 或者以 `sku_id % N` 作为分片路由

目标是：

1. 同一个热点 SKU 落在固定分区
2. 同一热点 SKU 的库存更新可以串行或低并发执行
3. 不同热点 SKU 之间仍可并行扩展

#### 消息体建议

建议库存削峰队列中的消息至少包含：

1. `request_id`
2. `user_id`
3. `order_token` 或 `pre_order_id`
4. `sku_id`
5. `quantity`
6. `biz_type`
7. `created_at`

#### 处理结果

库存削峰消费端需要明确三类结果：

1. 预占成功
2. 业务失败，例如库存不足
3. 系统失败，例如超时、数据库异常

并对这三类结果分别打点。

## 3.4 推进阶段

### 第一阶段：热点 SKU 白名单 + 局部排队试点

先不做全量异步，只对压测和监控已经确认的热点 SKU 启用。

阶段目标：

1. 限定改造范围
2. 验证队列化是否真的优于数据库直接竞争
3. 不影响普通订单路径

建议只在满足下面条件时启动这一阶段：

1. 多实例部署后，同一热点 SKU 的数据库竞争明显上升
2. 热点 SKU 场景的 `p95`、失败率或锁等待已经稳定高于普通 SKU
3. 当前同步稳定版本已经被充分固化，不再频繁改动

### 第二阶段：热点识别自动化

当白名单模式稳定后，再补：

1. 热点 SKU 动态识别
2. 热点阈值配置中心
3. 热点流量自动切换到异步受理

### 第三阶段：评估库存分片

只有当“按 SKU 排队”仍然扛不住时，才进一步考虑：

1. 库存分片表
2. 逻辑库存聚合
3. 更复杂的库存一致性对账

这个阶段属于更重的架构演进，不建议作为第一刀。

## 4. 订单异步化推进方案

## 4.1 先澄清“订单异步化”不等于“整个下单都丢 MQ”

订单异步化至少有三种强度不同的形态：

1. 后置动作异步化
2. 订单状态推进异步化
3. 热点订单受理异步化

对于当前项目，推荐按这个顺序推进。

## 4.2 第一阶段：后置动作异步化

这是最稳、最值得先做的一层。

同步链路中保留：

1. 下单校验
2. 库存预占
3. 订单主记录创建
4. 返回订单号和主状态

这里要特别强调：

1. `CreateOrder` 当前仍然保持同步闭环
2. 支付成功后的订单确认和库存确认扣减，当前也仍视为核心交易步骤
3. 第一阶段不改这两条主链路的返回语义

异步化的部分：

1. 站内消息
2. 审计日志
3. 用户通知
4. 统计埋点
5. 积分发放
6. 优惠券后处理

### 推荐方案：`Outbox + MQ`

在 `order-service` 和 `payment-service` 中引入 `outbox` 表：

1. 业务事务提交时，同时插入待投递事件
2. 后台投递器扫描 `outbox`
3. 投递到 `MQ`
4. 消费方异步处理

这样可以保证：

1. 主业务数据和待发送事件本地一致
2. MQ 短暂不可用时，消息不会丢
3. 后续扩展积分、券、消息中心时不会把主链路再拖重

### 这一阶段的前置建设

在真正消费这些事件之前，建议先把基础设施补齐：

1. `outbox` 表
2. 通用消息 envelope
3. 后台投递器
4. 消费重试与死信策略
5. 投递、消费、积压、死信指标

### 第一批事件建议

1. `order.created`
2. `order.cancelled`
3. `order.paid`
4. `payment.created`
5. `payment.succeeded`

## 4.3 第二阶段：热点订单局部异步受理

当热点 SKU 场景流量进一步放大，而且热点库存排队试点已经稳定后，才建议把“热点订单”从同步模式升级成“受理即返回，后台继续处理”。

推荐流程：

1. 用户请求到达 `gateway`
2. 系统识别为热点 SKU 请求
3. `order-service` 先写一条 `PENDING` 或 `ACCEPTED` 订单
4. 同事务写入 `outbox`
5. 后台把消息投递到热点订单队列
6. `order-worker` 异步执行库存预占与订单确认
7. 最终把订单状态推进成：
   - `CONFIRMED`
   - `FAILED`
   - `CANCELLED`

### 用户体验变化

这一步需要接受一个事实：

1. 热点订单不再保证请求内得到最终成功结果
2. 用户会先得到“排队中 / 已受理”
3. 后续要靠订单查询、轮询或通知拿最终状态

这类设计更适合：

1. 秒杀
2. 大促热点商品
3. 极端集中流量商品

不建议默认覆盖所有普通订单。

## 4.4 第三阶段：状态机化订单处理

如果后面进一步演进，可以把订单主链路沉淀成明确状态机，例如：

1. `PENDING`
2. `QUEUED`
3. `RESERVING_STOCK`
4. `RESERVE_FAILED`
5. `CONFIRMED`
6. `PAYING`
7. `PAID`
8. `CANCELLED`
9. `TIMEOUT_CLOSED`

这一步的价值是：

1. 支撑更复杂的异步链路
2. 让补偿和重试有清晰状态边界
3. 让监控和排障更可视化

但也会带来更高复杂度，因此不建议作为第一批落地内容。

## 5. 推荐的整体推进顺序

如果按照收益 / 风险比排序，建议按下面顺序推进。

### P0：先引入事件化基础设施

1. 增加 `outbox` 表
2. 增加通用投递器
3. 增加通用消息 envelope
4. 增加消息投递 / 重试 / 死信指标

当前进度：

1. `outbox` 表已在 `order-service`、`payment-service` 落地
2. 通用消息 envelope 已落地
3. 通用投递器和 Gorm store 骨架已落地
4. Kafka producer、指标打点、真实业务事件写入仍待继续推进

### P1：先做订单后置动作事件化

第一批只处理非核心后置动作：

1. `order.paid` 触发站内消息
2. `order.paid` 触发积分服务
3. `order.paid` 触发优惠券处理
4. `order.created` / `order.cancelled` 进入统计链路

目标是：

1. 不动主交易判定
2. 先把主链路之外的同步负担拿掉

### P2：热点库存局部排队化

这一步只在前面的事件基础设施稳定后，再针对明确热点 SKU 启动：

1. 引入热点库存队列
2. 增加 `inventory-worker`
3. 先基于热点 SKU 白名单启用

### P3：热点订单局部异步受理

在热点 SKU 场景验证库存排队稳定后，再推进：

1. 热点订单受理即返回
2. 后台确认库存和订单状态
3. 建立更完整状态机

### P4：更重的架构演进

这部分先后置：

1. 全量订单异步化
2. 库存分片
3. 更复杂的 Saga / 编排引擎

## 6. 关键技术点与治理要求

## 6.1 幂等

`MQ` 不是事务，重复投递和重复消费是常态。

因此必须保证：

1. 订单创建有稳定 `request_id`
2. 库存预占消费端幂等
3. 积分 / 优惠券 / 消息消费端幂等
4. 所有消费者都以业务主键去重

## 6.2 重试与死信

每个消费者必须区分：

1. 可重试失败
2. 不可重试业务失败
3. 超过重试上限进入死信

并补齐：

1. `retry_count`
2. `next_retry_at`
3. `dead_letter_reason`

## 6.3 监控与观测

`MQ` 方案要真正可运维，至少要补齐：

1. 生产成功率
2. 消费成功率
3. 消费延迟
4. 队列积压
5. 死信数量
6. 订单状态流转分布
7. 热点 SKU 排队等待时间

## 6.4 回查与补偿

订单异步化后，必须补齐：

1. 订单状态查询接口
2. 补偿任务
3. `outbox` 补投任务
4. 消费失败人工干预入口

## 7. 适合写进简历的工程亮点

如果这套方案落地，可以总结成下面几类简历亮点。

### 架构升级类

1. 基于 `Outbox + MQ` 重构订单后置动作分发链路，实现主交易数据与事件投递的一致性
2. 为热点 SKU 设计基于 `MQ` 的库存削峰队列，按 `sku_id` 分区串行化消费，降低数据库热点行竞争
3. 设计热点订单局部异步受理模型，将热点交易从同步下单升级为“受理即返回 + 后台确认”的状态机架构

### 高并发治理类

1. 基于多轮 `k6` 压测结果识别热点库存与订单主链路瓶颈，分阶段推进缓存、幂等、连接池、削峰排队等治理手段
2. 通过热点库存队列化和订单后置动作异步化，降低主链路同步步骤和热点写冲突，提升大促场景稳定性

### 工程治理类

1. 为消息生产、消费、死信、补偿、积压构建完整可观测体系
2. 为订单、库存、积分、券等跨服务异步链路建立统一幂等与重试规范

## 8. 当前建议

如果只保留一句话建议，那就是：

当前最值得做的，不是把 MeshCart 立刻全面消息化，而是先补齐 `outbox + MQ` 事件基础设施，把订单支付后的后置动作异步化；热点 SKU 的排队削峰能力作为下一阶段预留方案，在明确需要时再局部启用。

这样做的好处是：

1. 不会推翻已经验证稳定的同步交易基线
2. 能逐步把系统演进到更适合大规模集群和高并发活动的架构
3. 既有明确工程价值，也有很好的简历表达空间
