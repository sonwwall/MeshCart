# MeshCart 分布式事务与事件流程规范

## 1. 目的

本文档用于定义 MeshCart 后续涉及跨服务一致性、补偿、事件广播时的统一落地流程。

目标：

- 统一判断“当前问题是否需要分布式事务”
- 统一 `Saga`、本地事务、补偿、幂等、排障的实施步骤
- 统一 `Kafka` 的引入时机与职责边界
- 避免把“事务协调问题”误用消息队列处理
- 让后续新链路可以复用同一套评审、开发、测试、上线流程

本文档是流程规范。

- [distributed-transaction-design.md](./distributed-transaction-design.md)：描述当前商品创建 `P0` 问题的方案设计
- 本文档：定义以后所有类似问题的统一实施规则

## 2. 适用范围

当业务链路同时满足以下任一条件时，默认适用本规范：

- 一个写请求会修改两个或以上微服务的数据
- 调用方要求“要么整体成功，要么失败后可补偿”
- 某一步失败会留下对外不可解释的半成品数据
- 后续需要把主链路成功结果广播给多个下游系统

当前优先适用场景包括：

- 商品创建 + 库存初始化
- 订单创建 + 库存预占
- 支付成功 + 订单状态推进 + 库存确认扣减
- 核心写链路成功后的领域事件广播

## 3. 核心原则

### 3.1 先定义事务语义，再写代码

每条跨服务写链路在开发前必须先明确：

- 全局事务边界是什么
- 哪些步骤属于正向动作
- 哪些步骤需要补偿
- 事务成功后对外语义是什么
- 事务失败后允许留下什么数据，禁止留下什么数据

没有明确补偿语义的链路，不应直接进入实现阶段。

### 3.2 事务协调和事件广播是两类问题

- `Saga` 解决的是核心写链路一致性
- `Kafka` 解决的是事务成功后的事件传播与系统解耦

默认规则：

- 不要用 `Kafka` 替代当前应有的事务补偿设计
- 不要把“事件已发出”误当成“事务已收口”
- 先收口核心事务，再考虑把结果广播给更多下游

### 3.3 本地事务必须存在

即使引入分布式事务，每个参与方内部仍必须使用自己的本地事务保护单服务写入原子性。

### 3.4 幂等是强制要求

凡是分布式事务分支接口，必须支持：

- 正向幂等
- 补偿幂等
- 空补偿
- 防悬挂

没有幂等设计的分支接口，不允许接入协调器。

## 4. 技术选型规则

### 4.1 默认选 `Saga`

出现跨服务写一致性问题时，当前默认优先选 `Saga`。

适用条件：

- 参与方数量有限
- 每一步都有明确的补偿动作
- 当前业务不需要复杂预留态确认模型

### 4.2 暂不优先选 `TCC`

仅当业务天然需要 `Try / Confirm / Cancel` 三段状态，并且补偿删除已经不够表达业务语义时，再评估 `TCC`。

### 4.3 不自己实现协调器

默认要求使用成熟事务框架。

当前推荐：

- `DTM` 负责 `Saga` 协调

禁止事项：

- 在 `gateway` 或单个服务里手写一套长期维护的事务协调器

### 4.4 `Kafka` 的引入条件

只有出现以下信号时，才应该正式引入 `Kafka`：

- 一个事务成功后，需要通知多个独立下游
- 下游处理不适合继续同步 RPC 串行阻塞
- 需要领域事件回放、重试、削峰或消费者解耦
- 需要把事务成功结果传播给搜索、推荐、审计、风控、报表等外围系统

如果当前问题只是“两三个服务之间的写一致性”，默认先不要引入 `Kafka`。

## 5. 标准实施流程

后续所有分布式事务需求，按以下步骤推进。

### 5.1 需求评审

评审时必须先回答：

1. 谁是事务发起方
2. 哪些服务是事务参与方
3. 全局成功条件是什么
4. 每个分支的补偿动作是什么
5. 如果补偿失败，谁负责重试和排障
6. 是否需要后续事件广播

评审输出至少应包含：

- 事务边界
- 正向时序
- 失败补偿时序
- 幂等策略
- 状态隔离策略
- 排障入口

### 5.2 接口定义

每个分支至少应定义：

- 正向接口
- 补偿接口

接口入参默认要求带：

- `global_tx_id`
- `branch_id`

按业务补充：

- `operator_id`
- `biz_id`
- `sku_ids`
- `product_id`
- `order_id`

### 5.3 本地事务分支日志

每个参与方必须维护自己的事务分支日志表。

建议字段：

- `global_tx_id`
- `branch_id`
- `action`
- `biz_id`
- `status`
- `payload_snapshot`
- `error_message`
- `created_at`
- `updated_at`

建议状态：

- `prepared`
- `succeeded`
- `compensated`

该表用于：

- 正向幂等
- 补偿幂等
- 空补偿判断
- 防悬挂判断
- 事务排障

### 5.4 状态隔离

事务未完成前，不应让半成品数据以最终态暴露。

建议做法：

- 先落内部态或非最终可售态
- 全局事务成功后再切到目标态
- 事务失败后补偿删除，或明确转为失败态

### 5.5 协调器接入

当前标准做法：

1. `gateway` 或明确的应用服务作为事务发起方
2. 通过 `DTM` 注册正向分支和补偿分支
3. 协调器负责调用顺序、失败回滚、状态记录和重试
4. 发起方根据事务结果向调用方返回成功或失败

补充约束：

- 如果使用 `DTM workflow`
- 需要整体失败并触发补偿的错误，必须显式返回 `dtmcli.ErrFailure`
- 不能假设任意普通 RPC/network error 都会立即触发完整回滚

### 5.5.1 回滚触发规则

后续所有基于 `DTM workflow` 的链路，统一按以下规则实现：

1. 业务校验失败且应整体回滚时，返回业务错误并包装成 `dtmcli.ErrFailure`
2. 下游 RPC 不可达、超时、连接关闭等技术错误，如果业务语义也是“整笔事务失败”，同样包装成 `dtmcli.ErrFailure`
3. 只有“允许后续由协调器重试且不应立刻判定失败”的场景，才保留普通错误语义

如果不遵守这条规则，容易出现：

- 前序分支已经成功
- 当前分支失败
- 但事务没有立即进入补偿
- 从而留下脏数据窗口

### 5.5.2 回滚分支执行规则

补偿分支不是“注册了就一定执行”，而是要基于业务实际成功状态做保护。

统一要求：

- 只有分支正向动作已经确认成功，才允许执行该分支补偿
- 如果分支根本没有成功，回滚时必须跳过
- 跳过本分支补偿后，不能阻断更前面已成功分支的补偿

这条规则是为了解决以下真实问题：

- 某个中间分支因为服务下线导致正向 RPC 失败
- 如果它的补偿也继续打向同一个下线服务
- 会直接卡住回滚链
- 让前序已成功分支无法补偿

### 5.6 测试与验收

每条事务链路至少覆盖以下测试：

- 全部分支成功
- 中间分支失败并触发前序补偿
- 正向分支重复调用
- 补偿分支重复调用
- 空补偿
- 防悬挂
- 协调器重试下结果稳定
- 下游服务不可达时，未成功分支补偿被跳过且前序分支仍可回滚

验收标准：

- 不留下对外不可解释的脏数据
- 接口语义与真实落库结果一致
- 可以通过日志和事务 ID 快速定位失败分支

### 5.7 日志规范

分布式事务相关日志应当视为正式运维日志，而不是一次性调试输出。

最低要求：

- 发起方日志必须带 `gid`
- 分支日志必须带 `branch_id`
- 关键业务日志应带 `product_id`、`sku_ids`、`order_id` 等业务主键
- 失败日志应统一带 `code`、`message`、`error`

建议只保留以下主干事件：

- 全局事务 `start / succeeded / failed`
- 分支 `start / succeeded / failed`
- 补偿 `start / succeeded / failed / skipped`

不建议长期保留：

- 重复表达相同语义的日志
- 无事务上下文的 debug 输出
- 临时变量 dump

日志级别建议：

- `start / succeeded / skipped` 使用 `info`
- 可预期回滚起点可使用 `warn`
- 真正异常使用 `error`

## 6. Kafka 接入规范

### 6.1 Kafka 的职责边界

Kafka 只承担以下职责：

- 传播事务成功后的领域事件
- 向多个异步下游广播业务结果
- 削峰、解耦、异步重试

Kafka 不承担以下职责：

- 代替 `Saga` 做当前主写链路回滚
- 代替本地事务保证单服务写入原子性
- 在没有补偿语义的情况下“用消息最终一致”掩盖设计缺口

### 6.2 推荐接入位置

推荐顺序如下：

1. 先完成核心 `Saga`
2. 只在全局事务成功后发布领域事件
3. 由独立消费者订阅处理搜索、推荐、审计、通知等外围逻辑

### 6.3 事件发布约束

后续接入 Kafka 时，至少应保证：

- 事件只在事务成功后发布
- 事件有唯一业务键，支持消费幂等
- 消费失败可重试
- 不把外围消费者失败反向影响核心交易主链路

建议后续采用：

- 本地消息表或 Outbox 模式
- 消费者侧幂等表
- 统一事件命名和版本控制

## 7. 开发流程模板

后续新事务需求默认按下面模板执行：

1. 先补设计文档，明确事务边界、补偿语义、是否需要 Kafka
2. 评审通过后补 IDL 和状态流转定义
3. 增加事务分支日志表和 migration
4. 实现参与方正向接口
5. 实现参与方补偿接口
6. 接入协调器并改造发起方编排
7. 补齐幂等、空补偿、防悬挂测试
8. 如需事件广播，再补 Outbox/Kafka 发布链路
9. 更新 runbook、错误码、排障文档

## 8. 当前 P0 商品创建问题的落地口径

针对“商品创建成功但库存初始化失败”这个问题，当前按以下口径推进：

- 当前问题属于分布式事务，不属于 Kafka 优先解决的问题
- 当前主方案是 `Saga + DTM`
- `gateway` 作为事务发起方
- `product-service` 和 `inventory-service` 作为事务参与方
- 协调器具体实现采用 `DTM workflow`
- 商品补偿口径为删除本次事务创建的商品、SKU、SKU attrs
- 库存补偿口径为删除本次事务初始化的库存记录
- 事务未完成前，商品不应以最终可售态暴露
- 如果目标状态为 `online`，商品先以 `offline` 创建，库存成功后再执行最终上架
- 库存分支未成功时，回滚阶段跳过库存补偿，继续执行商品补偿

本次开发经验已经沉淀为以下通用规则：

- `DTM workflow` 中需要整体失败的错误必须显式返回 `dtmcli.ErrFailure`
- 回滚分支必须基于“正向是否真的成功”做保护，不能盲目执行补偿
- 排障时必须优先按 `gid` 联查发起方日志和参与方事务分支日志

本问题的详细方案设计见：

- [distributed-transaction-design.md](./distributed-transaction-design.md)

## 9. 文档同步要求

当后续新增或调整分布式事务链路时，至少同步以下文档：

- 本文档：如果影响通用流程规范
- 对应业务设计文档：如果影响具体链路语义
- [runbook.md](./runbook.md)：如果新增排障动作
- [error-code.md](./error-code.md)：如果新增错误码

如果本次开发过程中新增了以下任一经验，也必须回写本规范：

- 协调器错误语义约束
- 分支回滚顺序和跳过条件
- 新的幂等或空补偿处理规则
- 新的排障日志字段和查询入口
- 新的标准日志节点和日志级别口径

## 10. 当前结论

MeshCart 后续处理跨服务一致性问题时，统一遵循以下顺序：

1. 先判断是否属于核心写链路一致性问题
2. 如果是，优先使用 `Saga`
3. 用成熟协调器，不自己手写协调器
4. 每个参与方必须有本地事务和分支日志
5. 先收口事务，再考虑是否把成功结果通过 `Kafka` 广播出去

简化理解就是：

- `Saga` 负责把核心事务做对
- `Kafka` 负责把成功结果传播出去
