# 错误码规范

## 1. 目标

统一 MeshCart 项目错误码格式，满足以下要求：

- 网关与微服务错误可快速区分
- 错误可按模块快速定位
- 每个错误都有稳定的 `code` 与可读的 `msg`
- 便于日志检索、监控告警、前端提示

## 2. 编码格式

错误码固定为 **7 位数字**：

`TMMEEEE`

- `T`（第 1 位）：错误来源类型
  - `1`：网关错误（Gateway）
  - `2`：微服务错误（Service）
- `MM`（第 2-3 位）：模块编号
- `EEEE`（第 4-7 位）：模块内错误编号（0001-9999）

示例：

- `1010001`：网关-用户模块-参数错误
- `2010001`：用户服务-用户模块-用户不存在

## 3. 模块编号分配（建议）

| 模块编号(MM) | 模块名 |
|---|---|
| 00 | 通用 |
| 01 | 用户（user） |
| 02 | 商品（product） |
| 03 | 购物车（cart） |
| 04 | 订单（order） |
| 05 | 库存（inventory） |
| 06 | 支付（payment） |
| 07 | 营销（promotion） |
| 08 | 搜索（search） |

说明：

- 网关通用错误使用 `100xxxx`
- 某服务业务错误使用 `2 + 对应模块号 + xxxx`
- 新增模块时需先登记模块号，避免冲突

## 4. 错误定义分层

### 4.1 通用错误（跨服务）

放在：`app/common/errors.go`

建议包含：

- 通用参数错误
- 鉴权失败
- 无权限
- 资源不存在
- 系统内部错误

### 4.2 服务私有错误（按服务）

放在：`services/<service>/biz/errno/errors.go`

建议仅定义当前服务业务错误，例如：

- `services/user-service/biz/errno/errors.go`
- `services/order-service/biz/errno/errors.go`

## 5. 响应约定

所有对外响应（HTTP 或 RPC）应包含：

- `code`：机器可读错误码
- `message`：人类可读错误信息

建议：

- `message` 默认中文（面向国内业务）
- 内部日志保留详细技术错误，避免直接透出敏感信息

## 6. 使用规范

- 业务层（`biz/service`）返回业务错误对象（含 code/msg）
- 传输层（`rpc/handler`、`gateway/handler`）只做错误映射，不写业务判断
- 未知错误统一映射到模块内部错误（如 `xx9999`）或通用系统错误
- 同一业务场景使用稳定错误码，不随文案变动

## 7. 示例

### 7.1 网关错误示例（T=1）

- `1000001`：网关通用参数错误
- `1000002`：网关鉴权失败
- `1010001`：网关用户接口参数错误

### 7.2 微服务错误示例（T=2）

- `2010001`：用户不存在
- `2010002`：用户名或密码错误
- `2030001`：购物车不存在
- `2040001`：订单不存在

## 8. 变更流程

新增错误码时按以下步骤执行：

1. 确认 `T/MM` 是否正确
2. 在对应文件新增常量与错误变量
3. 在本文件补充错误码说明（可按模块追加）
4. 在接口文档中标注该错误码

## 9. 禁止事项

- 禁止不同错误复用同一个错误码
- 禁止业务成功返回非 0 错误码
- 禁止在 handler 中硬编码大量错误码（应引用 errno 常量）

## 10. 与当前代码对应

当前建议落点：

- 通用：`app/common/errors.go`
- 用户服务：`services/user-service/biz/errno/errors.go`

后续新增服务时按同样模式扩展。
