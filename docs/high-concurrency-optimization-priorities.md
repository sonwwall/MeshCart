# MeshCart 高并发优化优先级建议

本文档基于当前项目前 8 轮 `k6` 压测结果整理，目标不是讨论所有可能的高并发技术方案，而是回答一个更实际的问题：

- 现在这个系统，应该优先优化哪里
- 哪些方案值得尽快做
- 哪些方案可以先不急

原始压测过程与逐轮记录见 [k6-loadtest-plan.md](/Users/ruitong/GolandProjects/MeshCart/docs/k6-loadtest-plan.md)，瓶颈归纳见 [loadtest-bottleneck-analysis.md](/Users/ruitong/GolandProjects/MeshCart/docs/loadtest-bottleneck-analysis.md)。

## 1. 结论先行

基于当前压测结果，最值得优先优化的不是整套“高并发大架构”，而是交易写链路本身，尤其是：

1. 订单创建链路
2. 库存预占链路
3. 与库存和订单相关的数据库事务范围

当前不建议一上来就把重点放在：

1. 大规模引入缓存来“解决所有问题”
2. 用消息队列替代核心交易同步链路
3. 直接把订单创建做成完全异步

原因很简单：

- 读链路目前整体稳定
- 写链路才是当前最敏感的部分
- 主问题更像是同步写路径的稳定性和事务成本，而不是读吞吐先扛不住

## 2. 当前系统表现出的真实特征

### 2.1 读链路整体稳定

前几轮压测里：

- 商品列表
- 商品详情

都没有表现出明显的性能崩点。即使在接近 `100 req/s` 的量级下，读链路的 `p95` 也仍然较低。

这意味着当前阶段：

- `gateway`
- `product-service`

主读路径不是系统最急迫的瓶颈。

### 2.2 订单写链路最敏感

多轮压测里，`POST /api/v1/orders` 是最容易出现 RT 抬升的接口。

虽然高延迟没有在每一轮都稳定复现，但订单链路反复表现出：

- 对环境波动敏感
- 对库存状态敏感
- 在中等并发下更容易出现明显 RT 放大

所以当前最值得优先投入优化资源的，不是商品读接口，而是订单创建主链路。

### 2.3 库存预占是当前最可疑的一段

已有结果说明，订单链路一旦变慢，最值得怀疑的是：

`order-service -> inventory-service reserve_sku_stocks -> MySQL`

这一段当前最像“写链路放大器”。

### 2.4 测试数据状态会污染结论

后几轮压测反复证明了一件事：

- 热点 SKU 或普通 SKU 如果被打空
- 压测结果就会混入“库存不足”这类业务失败

这不是性能瓶颈本身，但它会让后续分析失真。

因此从工程角度看，“压测前库存检查和重置”也是高并发治理的一部分。

## 3. 立即优先级最高的优化项

### 3.1 优化库存预占链路

这是当前优先级最高的优化点。

建议重点检查：

1. 库存预占事务是否过长
2. 是否存在不必要的“先查再改”
3. 是否存在一次下单触发多次库存相关读写
4. 热点 SKU 更新语句是否命中合适索引
5. 是否存在会放大锁竞争的更新模式

优化目标：

- 减少库存预占 SQL 的执行时间
- 缩短事务持有时间
- 降低热点 SKU 下的竞争放大

### 3.2 优化订单创建同步主流程

当前订单创建链路里，任何一个同步步骤变慢，都会直接体现在用户请求 RT 上。

因此建议优先把同步主流程收缩到最小必要集合。

更合理的同步主链路应尽量只保留：

1. 参数校验
2. 商品 / SKU 有效性确认
3. 库存预占
4. 订单主记录写入
5. 返回订单号

不应该继续停留在主链路里的内容，包括但不限于：

1. 非关键日志扩展写入
2. 运营统计类写入
3. 非必要的审计记录
4. 可以通过事件补偿的后置动作

### 3.3 优化订单与库存相关 SQL 和索引

在没有数据库层证据前，不建议拍脑袋重构架构，但非常建议先检查数据库基础项：

1. 订单表主写 SQL 是否存在多余查询
2. 库存表是否具备覆盖热点更新路径的索引
3. 预占记录表是否存在低效扫描
4. 幂等记录表是否存在写放大

如果 SQL 和索引本身没收紧，再上缓存或 MQ，通常只是把问题包起来，不是解决问题。

## 4. 应该尽快做，但不是第一刀的优化项

### 4.1 把非核心写入异步化

异步化是值得做的，但要有边界。

适合优先异步化的内容：

1. 审计日志
2. 事件通知
3. 统计聚合
4. 报表类写入
5. 非核心状态广播

这样做的价值是：

- 缩短订单主请求的同步路径
- 把不影响用户主结果的动作从关键链路移走

但要强调：

- 异步化不等于把主交易一致性丢掉
- 核心订单创建与库存预占不应该为了“快”而直接改成纯异步

### 4.2 强化幂等与动作记录模型

订单和支付已经有一定幂等保护，这方向是对的，但仍然值得继续优化。

建议检查：

1. 幂等记录是否写入过多
2. 是否存在多次状态写入才能完成一次请求
3. 是否存在“插入一条、更新一条、再查一条”的冗余路径

这类优化对 RT 的收益可能不像缓存那样直观，但对高并发下的稳定性很重要。

### 4.3 固化压测前数据重置流程

这不是业务优化，但它会直接影响后续性能判断的准确性。

建议补齐：

1. 压测前库存检查
2. 自动补库存
3. 热点 SKU 与普通 SKU 的固定数据池
4. 每轮压测后的状态回收或重新准备

如果这个流程不稳定，后续很多“瓶颈”其实都只是样本脏了。

## 5. 可以考虑引入缓存的部分

### 5.1 商品详情和商品列表缓存

这部分是当前最适合做缓存的区域。

原因：

1. 数据天然偏读多写少
2. 当前接口形态适合做热点缓存
3. 即便当前不是主瓶颈，缓存也能降低下游读压力

适合缓存的对象：

1. 商品详情
2. SKU 基础信息
3. 热门商品列表

但要明确：

- 这是读优化
- 不是当前写链路瓶颈的核心解法

所以缓存可以做，但不建议排在库存预占和订单事务优化之前。

### 5.2 不建议简单把库存判断做成缓存优先

库存属于强一致敏感数据。

当前阶段不建议直接走这种思路：

- 先查 Redis
- 再异步落库
- 再靠补偿修一致性

因为这会显著增加一致性复杂度。

除非后面明确要做更重的秒杀/抢购型架构，否则现在不应把库存缓存化当成第一选择。

## 6. 可以考虑引入消息队列的部分

### 6.1 MQ 适合承接订单后的非核心事件

消息队列当前更适合做“同步链路瘦身”，而不是直接承载核心交易一致性。

适合接入 MQ 的场景：

1. 下单成功后的事件广播
2. 支付成功后的后续处理
3. 运营通知
4. 风控埋点
5. 报表和统计消费

这样可以把主链路里不必要的同步动作移出去。

### 6.2 不建议用 MQ 直接替换核心同步交易路径

当前不建议做这种改造：

1. 下单请求先入 MQ 再慢慢创建订单
2. 库存预占完全异步执行
3. 支付创建依赖异步最终一致返回

原因：

1. 会增加用户侧不确定性
2. 会明显提升补偿复杂度
3. 当前压测结果还不足以证明必须走到这一步

换句话说，MQ 现在更适合作为“减负工具”，而不是“主交易重构工具”。

## 7. 异步入库适合哪些地方

异步入库不是不能做，但必须分清核心数据和附属数据。

适合异步入库的：

1. 操作日志
2. 审计日志
3. BI / 分析型数据
4. 统计快照
5. 非关键事件记录

不适合当前直接异步化的：

1. 订单主记录
2. 库存预占主记录
3. 支付主状态记录

如果把这些核心数据异步入库，只会把延迟问题换成一致性风险。

## 8. 推荐的优化顺序

如果按投入产出比排序，建议按下面顺序推进。

### 8.1 第一阶段：立刻做

1. 补数据库层观测
2. 优化库存预占 SQL / 事务
3. 收缩订单创建同步主流程
4. 固化压测前库存检查和补库存流程

### 8.2 第二阶段：中期做

1. 异步化非核心写入
2. 精简幂等记录和状态写入路径
3. 给商品详情、商品列表增加缓存

### 8.3 第三阶段：架构演进时再做

1. 更完整的 MQ 事件化
2. 面向极端热点场景的库存架构升级
3. 更复杂的缓存预扣与异步对账体系

## 9. 当前最不建议的误区

### 9.1 不要因为“高并发”三个字就先上全套中间件

当前阶段最危险的误区不是技术不够，而是过早复杂化。

如果现在直接上：

1. 全量 MQ 化
2. 全链路异步化
3. 缓存覆盖所有核心状态

你会得到更复杂的系统，但不一定得到更高的吞吐和更稳的 RT。

### 9.2 不要把所有问题都归因于缓存没加

当前压测结果没有显示读链路先出问题，因此“先上缓存救系统”不是最合理的优先级。

缓存应该服务于：

- 降低读压力

而不是被当成：

- 写链路延迟异常的万能答案

### 9.3 不要在没有数据库证据前直接重构交易模型

当前最该补的是证据，而不是先改一大轮架构。

如果数据库层最终证明：

- 锁竞争明显
- 慢 SQL 集中
- 某类事务过长

那就有充分理由推进更深层的改造。

否则很容易出现“改了很多，但没打中根因”的情况。

## 10. 最终建议

如果只保留一句话建议，那就是：

当前最该优化的是订单创建和库存预占这条同步写链路本身，而不是急着用缓存、消息队列或异步化去覆盖问题。

更具体地说：

1. 先把库存预占做短、做稳
2. 先把订单主链路同步步骤做薄
3. 先把数据库观测补齐
4. 再把非核心写入异步化
5. 最后再考虑读缓存和更重的架构演进

这条路线更符合当前 8 轮压测暴露出来的真实问题，也更符合工程投入产出比。
