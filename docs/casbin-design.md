# Casbin 权限设计方案

## 1. 文档目标

本文档描述 MeshCart 当前基于 Casbin 的权限设计，目标是提供一套可被多个业务模块复用的授权方案，而不是只服务于商品模块。

本文档覆盖以下内容：

- 为什么引入 Casbin
- 当前权限模型
- 角色与资源的抽象方式
- 在网关中的落点
- 商品模块的具体应用方式
- 后续如何复用到其他模块

## 2. 设计目标

当前权限系统需要解决两个问题：

- 把“是否有权限访问某个资源”从业务逻辑中抽离出来，避免每个模块各写一套 if/else
- 支持“角色权限 + 资源归属”的组合判断，而不是只有简单的角色判断

对当前项目而言，这意味着：

- 普通用户只能访问公开资源
- 管理员可以访问管理接口
- 但管理员也不是全局无限权限，仍然要受到资源归属约束

## 3. 为什么使用 Casbin

当前项目如果继续手写权限判断，很快会出现这些问题：

- 权限条件分散在 handler / logic 各处，不容易统一维护
- 同一个角色在不同模块下的权限规则容易写出不一致行为
- 后续如果加运营、审核、商家等角色，会迅速失控

Casbin 适合当前阶段的原因：

- 可以把“角色、资源、动作”的判定抽象成统一规则
- 可以保留 Go 代码内嵌模型，不一定一开始就引入外部策略存储
- 后续如果权限模型复杂化，可以逐步演进到持久化策略

## 4. 当前落点

当前实现位置：

- `gateway/internal/authz/casbin.go`

当前设计选择：

- Casbin 只放在 `gateway` 层
- 由 `gateway` 负责做 HTTP 访问控制
- 下游 `product-service` 仍然专注商品数据读写，不直接感知 Casbin

这样做的原因：

- 网关天然是权限入口
- 避免每个微服务各自维护一套授权规则
- 可以先把“对外授权”统一在一个地方，再决定是否把内部 RPC 也纳入授权

## 5. 权限模型

### 5.1 当前角色

当前定义四种角色：

- `guest`
- `user`
- `admin`
- `superadmin`

当前角色来源：

- 未登录请求：`guest`
- 已登录用户：根据 `user-service` 返回的 `role` 决定
- 当前角色保存在用户表 `users.role`
- 迁移已有用户数据时，系统会把最早注册的用户提升为初始 `superadmin`
- 全新空库场景下，首个注册用户会自动成为 `superadmin`

说明：

- 当前角色已经迁移到用户域统一管理
- 网关不再通过启动参数维护管理员名单
- JWT 中的 `role` 是登录时生成的角色快照

### 5.2 目标角色设计

当前三角色模型可以支撑基础联调，但长期建议演进为：

- `guest`
- `user`
- `admin`
- `superadmin`

各角色建议职责如下：

- `guest`：未登录访客，只能访问公开资源
- `user`：普通登录用户，只能访问自己的用户域资源和公开资源
- `admin`：业务管理员，可以创建资源、管理自己创建的业务数据
- `superadmin`：平台级管理员，负责角色分配、权限治理、跨资源管理

当前不建议让 `admin` 直接拥有平台级提权能力，原因是：

- 商品管理和权限管理是两类职责
- 如果 `admin` 同时能改别人角色，权限边界会迅速失控
- 后续做审计时，也需要明确谁在做平台治理动作

因此更合理的职责划分是：

- `admin`：管理业务资源
- `superadmin`：管理角色和高权限操作

### 5.3 资源

当前 Casbin 把受控对象抽象为：

- `product`

后续可以继续扩展：

- `order`
- `inventory`
- `user`
- `admin_panel`

### 5.4 动作

当前商品模块使用四类动作：

- `read_online`
- `read_private`
- `write_own`
- `create`
- `manage_role`

语义如下：

- `read_online`：读取公开商品
- `read_private`：读取非公开商品，但要求资源归属当前用户
- `write_own`：修改自己创建的商品
- `create`：创建商品
- `manage_role`：修改用户角色

### 5.5 请求属性

当前 Casbin matcher 使用以下请求属性：

- `sub`：角色
- `obj`：资源名
- `act`：动作
- `owner`：资源所有者
- `uid`：当前用户 ID
- `status`：资源状态

这套设计的重点是：

- 不只看角色
- 还把资源归属和资源状态一起纳入判定

## 6. 当前策略

当前内嵌策略如下：

- `guest` 可以 `read_online product`
- `user` 可以 `read_online product`
- `admin` 可以 `read_online product`
- `admin` 可以 `read_private product`
- `admin` 可以 `write_own product`
- `admin` 可以 `create product`
- `superadmin` 可以 `read_online product`
- `superadmin` 可以 `read_private product`
- `superadmin` 可以 `write_own product`
- `superadmin` 可以 `create product`
- `superadmin` 可以 `manage_role user`

这代表当前权限含义：

- 所有人都能看 `online` 商品
- 只有管理员能看非公开商品
- 但管理员也只能看自己创建的非公开商品
- 只有管理员能创建商品
- 只有管理员能修改自己创建的商品
- `superadmin` 可以管理全量商品
- 只有 `superadmin` 可以修改用户角色

## 7. 商品模块中的应用

### 7.1 商品列表

规则：

- 未登录或普通用户：只能查 `status=online`
- 管理员：如果查 `draft/offline`，会自动带上 `creator_id = 当前用户`

这意味着管理员也不能查询其他管理员创建的草稿商品。

### 7.2 商品详情

规则：

- `online` 商品：公开可见
- `draft/offline` 商品：只有创建者本人且具备管理员角色时可见

如果当前用户没有权限，网关不暴露“存在但无权访问”的细节，而是按不可访问处理。

### 7.3 创建商品

规则：

- 只有管理员可创建
- 创建时由网关把当前登录用户写入 `creator_id`

### 7.4 更新商品

规则：

- 只有管理员可更新
- 且只能更新自己创建的商品
- 更新时由网关把当前用户写入 `operator_id`

### 7.5 修改商品状态

规则：

- 只有管理员可修改商品状态
- 且只能修改自己创建的商品

### 7.6 修改用户角色

规则：

- 只有 `superadmin` 可访问
- 当前已落地接口：`PUT /api/v1/admin/users/:user_id/role`
- 角色修改由 `user-service` 落库
- 不允许把最后一个 `superadmin` 降级，避免系统失去平台管理员

## 8. 数据模型配合

为了支持“资源归属”判断，商品表增加了以下字段：

- `creator_id`
- `updated_by`

作用：

- `creator_id`：标记商品是谁创建的
- `updated_by`：标记最后一次更新操作由谁执行

这两个字段并不是为了前端展示，而是为了权限控制和审计。

## 9. JWT 配合

当前 JWT claims 除了原有字段外，新增：

- `role`

当前登录流程：

1. 用户登录成功
2. `user-service` 返回用户当前角色
3. 网关把 `role` 写进 JWT
4. 后续商品管理接口和用户角色管理接口使用该角色参与 Casbin 判定

说明：

- 角色修改后，旧 JWT 不会自动刷新
- 如果用户刚被提权或降权，需要重新登录以获取新角色

## 10. 下一阶段设计方案

当前已经完成从 `ADMIN_USER_IDS` 到用户域角色落库的迁移，下一阶段重点是完善角色管理和审计能力。

### 10.1 设计原则

- `gateway` 不直接连接数据库
- 用户身份和角色归属放在 `user-service`
- `gateway` 只做认证、鉴权编排和策略执行
- Casbin 继续作为统一授权引擎，但角色来源改为用户域

这意味着：

- 不需要给 `gateway` 新建数据库
- 不建议让 `gateway` 直接查用户表
- 角色数据应保存在 `meshcart_user` 所属用户库中

### 10.2 角色存储方案

下一阶段建议优先采用简单可落地的方式：

- 在用户表增加 `role` 字段

推荐枚举：

- `user`
- `admin`
- `superadmin`

如果后续权限复杂化，再进一步拆成：

- `roles`
- `user_roles`
- `permissions`

当前项目规模下，先在用户表增加 `role` 字段更实用，迁移成本也更低。

### 10.3 登录与鉴权链路

目标链路如下：

1. 用户通过 `user-service` 登录
2. `user-service` 返回用户基础信息和角色
3. `gateway` 使用该角色签发 JWT
4. 后续请求由 `gateway` 从 JWT 中解析 `role`
5. Casbin 基于 `role + obj + act + owner + status` 做授权判定

在这套设计下：

- 角色来自数据库，而不是启动参数
- JWT 只是角色快照，不是角色真源
- 如果角色被修改，可以通过重新登录、强制刷新 token 或版本化 token 机制生效

### 10.4 `superadmin` 职责

建议新增 `superadmin`，但职责应当收敛在平台治理层，而不是所有业务动作都直接放开。

`superadmin` 建议具备：

- 修改用户角色，例如 `user -> admin`
- 回收管理员权限，例如 `admin -> user`
- 查看角色分配结果
- 查看关键权限变更记录
- 必要时管理全部商品和其他业务资源

`superadmin` 不建议承担：

- 普通商品录入日常操作
- 所有业务流程默认亲自参与

原因是平台治理动作和日常业务操作最好分层。

### 10.5 管理接口建议

如果引入 `superadmin`，建议后续增加用户角色管理接口，例如：

- `GET /api/v1/admin/users`
- `PUT /api/v1/admin/users/:user_id/role`

其中：

- 只有 `superadmin` 可以访问
- 请求会写入权限变更审计日志
- 角色修改后应明确生效方式，例如要求重新登录

### 10.6 与商品模块的配合

在引入 `superadmin` 后，商品模块建议采用两层规则：

- `admin`：只能管理自己创建的商品
- `superadmin`：可以管理全量商品，或者在过渡期仍然遵循 owner 规则，由业务配置决定

更推荐的长期做法是明确区分：

- 业务管理员权限
- 平台治理权限

而不是让所有管理员共享同一套能力。

### 10.7 迁移步骤建议

建议按以下顺序迁移：

1. 在 `user-service` 的用户表增加 `role` 字段
2. 登录接口和个人信息接口返回 `role`
3. `gateway` 登录时改为使用 `user-service` 返回的角色签发 JWT
4. 增加 `superadmin` 管理接口
5. 为角色变更补审计日志
6. 再考虑把 Casbin policy 做持久化

这样可以避免一次性切换导致所有环境同时失效。

## 11. 为什么不把权限全写在商品服务里

当前不把 Casbin 放进 `product-service`，主要因为：

- HTTP 访问控制天然属于网关职责
- 商品服务应该优先关注业务数据本身
- 如果每个服务都自己接 Casbin，短期内会造成权限规则分散

但这不代表下游服务永远不需要自保护。

后续如果出现：

- 多个入口直接调用同一商品 RPC
- 管理后台绕过网关直连内部服务
- 内部服务之间也要做细粒度授权

那就需要进一步评估在服务侧补充授权或签名校验。

## 12. 可复用设计原则

这套 Casbin 设计在其他模块复用时，建议遵循以下原则：

- 角色统一由用户身份系统或网关统一解析
- 资源统一抽象成 `obj`
- 动作统一抽象成 `act`
- 资源归属、状态等动态属性作为请求参数参与 matcher
- 业务模块不要直接写死“管理员万能”的假设

举例：

- 订单模块可以定义 `read_own_order`、`cancel_own_order`
- 库存模块可以定义 `adjust_own_store_inventory`
- 营销模块可以定义 `publish_campaign`、`read_private_campaign`

## 13. 当前局限

当前实现仍有几个明显限制：

- Casbin 策略写死在代码里，还没有持久化
- 当前商品模块和用户角色管理已接入授权
- 还没有操作审计日志和权限变更审计
- 角色变更后仍依赖重新登录刷新 JWT

这些限制在当前阶段是可接受的，但不适合长期扩展。

## 14. 后续演进方向

建议的下一步演进如下：

- 完善 `superadmin` 的角色管理与审计能力
- 把 Casbin policy 从代码内嵌迁移到可配置存储
- 引入更细的角色，例如运营、审核、商家
- 把“只可操作自己创建的资源”扩展为更通用的 owner / tenant / organization 模型
- 为关键权限行为补操作审计和权限审计

## 15. 角色演进建议

### 14.1 当前为什么只保留 3 个角色

当前只保留以下角色：

- `guest`
- `user`
- `admin`

原因是当前项目还处在基础能力搭建阶段：

- 用户侧只需要区分“是否登录”
- 管理侧只需要区分“是否能操作商品”
- 继续细分角色虽然看起来完整，但短期只会增加维护成本

所以当前三角色模型是够用的，重点先把“公开访问”和“资源归属约束”做对。

### 14.2 下一阶段建议新增的角色

如果商品管理后台继续演进，我建议优先新增两类角色：

- `operator`
- `reviewer`

其中：

- `operator`：偏运营角色，可以创建和编辑商品，但不一定能做高风险发布动作
- `reviewer`：偏审核角色，可以查看待审核商品、执行发布或驳回，但不一定负责日常编辑

这样做的原因是当前 `admin` 实际上混合了多种职责：

- 创建商品
- 编辑商品
- 查看非公开商品
- 修改商品状态
- 未来可能还会做审核和发布

这些能力长期都放在 `admin` 下，会让角色越来越粗，边界越来越模糊。

### 14.3 多商家场景建议新增的角色

如果后续项目进入多商家或多租户场景，建议引入更明确的租户角色：

- `merchant_admin`
- `merchant_operator`

语义可以是：

- `merchant_admin`：商家管理员，管理本租户内的商品和运营动作
- `merchant_operator`：商家运营，只能做日常编辑，权限比 `merchant_admin` 更窄

届时权限判断将不再只是“是否本人创建”，而会逐步演进到：

- 是否属于同一租户
- 是否属于同一组织
- 是否在当前资源授权范围内

### 14.4 建议的演进顺序

我建议按以下顺序扩角色，而不是一次性全部引入：

1. 当前阶段：保留 `guest` / `user` / `admin` / `superadmin`
2. 管理后台细分阶段：新增 `operator` / `reviewer`
3. 多商家阶段：新增 `merchant_admin` / `merchant_operator`
4. 平台化阶段：再考虑更细的组织、审批、跨团队角色

这样可以避免权限模型在业务尚未稳定时过度设计。
